# Playsync V2 작업 보드 (tickets)

> 브랜치 하나 = 티켓 하나 = PR 하나.
> 각 티켓은 [fixlist.md](./fixlist.md)의 항목을 참조만 한다. 문제의 상세와 수정
> 코드는 fixlist에 있고, 여기에는 **무엇을 왜 묶었는지와 순서**만 적는다.
>
> fixlist는 심각도 순으로 정렬된 **발견 대장**이고, 이 문서는 **작업 순서**다.
> 둘이 다른 이유는 커플링 때문이다 — 같은 줄을 고치는 항목은 심각도가 달라도
> 한 브랜치에서 함께 처리해야 한다.

## 순서를 정한 기준

1. **커플링 우선** — 같은 함수/같은 줄을 고치는 항목은 반드시 한 티켓.
2. **의존성** — 다른 티켓의 산출물을 쓰는 티켓은 뒤로.
3. **그다음 심각도** — 위 둘이 같으면 P0 → P1 → P2 → P3.
4. **첫 티켓은 인프라 없이 도는 것으로** — 브랜치/테스트/PR/CI 루프를 처음
   도는 데 Redis·DB 셋업이 끼면 루프가 아니라 환경 씨름이 된다.

## 테스트 방침

fixlist의 [P3-4]는 티켓이 아니라 **다른 티켓들에 녹는다**. 거기 나열된 케이스 9개 중
6개가 T1/T4의 회귀 테스트이므로, 각 티켓에서 실패하는 테스트를 먼저 쓰고 고치면
자연히 생긴다. T13에는 어디에도 안 붙는 케이스(사이드팟, 환급)만 남는다.

## T2 — 게임 상태 동시성

**항목**: P0-4, P1-3
**범위**: `redis.service.ts`(락 헬퍼 추가), `playsync.service.ts`, `dealer.service.ts`
**프론트 영향**: 없음

`getSnapShot → 수정 → saveSnapShot`이 락 없이 4곳에서 동시 실행 가능 (lost update).

**왜 함께 묶는가**: fixlist [P0-4] 수정 2번이 "[P1-3] 수정판과 함께 적용"이라고
명시한다. 락으로 감싸는 것과 턴 재검증이 `handleAction` 같은 본문이라 따로 못 고친다.
게다가 [P1-3]에는 **순서 함정**이 있다 — 턴 재검증을 타임아웃 잡 제거보다 **먼저**
해야 한다. 뒤에 두면 다음 플레이어의 타이머를 유실시켜 P1-1 데드락이 다른 경로로
재발한다. fixlist의 원안이 이 결함을 가지고 있었고 2차 검증에서 교체되었다.

**왜 T1 다음인가**: 나머지 티켓이 상태를 만지는 동안 lost update가 살아 있으면
버그의 원인이 내 수정인지 레이스인지 구분이 안 된다. 바닥을 먼저 깐다.

**인지하고 갈 트레이드오프**: `resolveWinners`는 리바인 응답을 최대 15초 기다리는
동안 락을 쥔다. I/O 대기를 락 안에 넣는 것은 원래 안티패턴이다. 지금은 TTL 30초로
넘기고, 리바인 대기를 락 밖으로 빼는 리팩터는 T5 이후 별도로 다룬다.

**완료 조건**
- [x] `withTableLock` 헬퍼 — `SET NX PX` 획득, 토큰 대조 후 Lua로 해제
- [x] 상태를 수정하는 4곳(`handleAction`, `startPreFlop`, `handleDealerAction`, `resolveWinners`) 전부 적용
- [x] 턴 재검증이 타임아웃 잡 제거보다 먼저 실행됨
- [x] KICKED 유저 처리에서 `engine.act` 이중 호출 제거

### 작업 중 추가로 나온 것 (PR #3)

fixlist에 없던 항목들. 락만으로는 안 닫히는 구멍이라 함께 처리했다.

- **`actionDeadline`을 서버가 읽는다.** 백엔드 4곳이 쓰기만 하고 읽는 곳은 프론트
  카운트다운뿐이었다. "30초 지났나"를 판정하는 주체가 BullMQ 잡의 발화 시각밖에
  없었던 셈 — 시간 기반 규칙을 이벤트 도착 순서로 대체한 구조였다. 마감이 지나
  도착한 액션은 도착 순서와 무관하게 `TIME_OUT`으로 처리한다.
- **타이머 세대(`timerEpoch`).** `jobId`가 `tableId` 고정이라, 제거에 실패한 상태에서
  `add`하면 BullMQ가 조용히 무시하고 그 잡이 `removeOnComplete`로 사라져 아무도
  타이머가 없는 테이블이 남았다. 진행 중인 잡은 지울 수 없고 BullMQ는 at-least-once라
  제거 성공에 의존하는 구조 자체가 틀렸다. `jobId`에 세대를 붙여 충돌을 없애고,
  낡은 잡은 세대 불일치로 스스로 폐기되게 했다.
- **`startPreFlop`도 같은 순서 결함이 있었다.** phase를 확인하기 전에 잡부터 지우고
  조기 반환해서, 진행 중인 핸드의 타이머가 통째로 사라졌다.
- **큐 교체를 `scheduleTurnTimeout` 한 곳으로 모았다.** 조기 반환 경로가 큐를 아예
  부를 수 없으므로, 순서가 다시 뒤집히는 것이 주석이 아니라 구조로 막힌다.
- 테이블에 없는 유저(`playerIdx === -1`)를 진입점에서 거부. 잡 delay 31000 /
  deadline 30000 불일치 정리.

**남긴 부채**: `resolveWinners`의 리바인 대기(최대 15초)가 여전히 락 안에 있다.
TTL 30초로 넘겼다. → T5 (PR #6에서 해소, 정산을 3구간으로 분해)

## T3 — WS 인바운드 경계 + contract 도입

**항목**: P0-2, P0-3, P3-2, N-9.3
**범위**: `ws.gateway.ts`, `packages/contract`, `frontend`(액션 전송부)
**프론트 영향**: **있음** — 유일하게 백/프론트 동시 변경

게이트웨이의 인바운드 경계 전체를 한 번에 세운다.

- P0-2: `data.action`을 검증 없이 엔진에 전달. `ActionType`이 숫자 enum이라
  유저가 `TIME_OUT`/`DEALER_KICK`/`DEALER_FOLD`를 직접 보낼 수 있다.
- P0-3: 딜러 JWT의 `tableId`와 접속 쿼리의 `tableId`를 대조하지 않는다.
- N-9.3: 인증된 유저가 아무 `tableId`로나 붙어 상태를 수신한다 (관전).
- P3-2: Origin 미검증. `@WebSocketGateway({ cors })`는 `WsAdapter`에서 무시되므로
  핸드셰이크의 `Origin` 헤더를 직접 봐야 한다.

**왜 함께 묶는가**: 넷 다 `handleConnection` / `handlePlayerAction` 두 함수에 들어간다.
따로 하면 같은 함수를 네 번 고치고 그때마다 프론트 회귀 확인을 반복한다.

**왜 contract가 여기서 도입되는가**: P0-2를 화이트리스트 배열로 막으면 액션이
추가될 때 배열 갱신을 잊는 순간 다시 뚫린다. contract에 클라가 보낼 수 있는 액션만
정의하면 **타입이 곧 화이트리스트**가 되어 그 실수가 불가능해진다. 즉 이 티켓에서
`packages/contract`의 첫 실제 내용이 채워진다.

**숫자 enum → 문자열**: 같이 한다. 와이어에 `4` 대신 `"CALL"`이 흐르면 로그가 읽히고
오프바이원 한 칸으로 다른 액션이 되는 버그 클래스가 사라진다. 프론트 액션 전송부도
같이 바뀐다 — UI/UX는 나중에 갈아엎지만 **이 변경은 로직이라 그대로 남는다.**

**완료 조건**
- [x] `PlayerActionSchema` (`.strict()`) — CHECK/CALL/FOLD/RAISE만 존재
- [x] 딜러 토큰 `tableId` 대조
- [x] 접속 시 테이블 소속 검증
- [x] Origin 검증, 오해를 부르는 `cors: true` 제거
- [x] 프론트가 contract 스키마로 액션 전송
- [x] `handlePlayerAction`에 role 검사 (딜러 토큰으로 플레이어 액션 차단)

### 작업 중 추가로 나온 것 (PR #4)

- **딜러 경로도 스키마가 필요했다.** `DealerActionSchema`를 함께 정의했다. 모르는
  액션이 오면 switch가 아무 case에도 안 걸려 `updatedState`가 `undefined`인 채로
  테이블 전원에게 브로드캐스트되고 있었다. 스키마 + `if (updatedState)` 가드로 막았다.
- **contract가 즉시 죽은 코드 두 개를 드러냈다.** 프론트가 `token`/`tableId`를
  페이로드에 실어 보내는데 서버는 읽지 않았고(핸드셰이크에서 이미 인증됨),
  ALL-IN TO CALL 버튼이 읽히지 않는 `amount`를 보내고 있었다. `.strict()`가
  이것들을 에러로 만들어서 지웠다 — 리팩터가 아니라 검증이 찾아낸 것.
- **`ActionType` 숫자 enum → 문자열을 엔진까지 관통시켰다.** contract가 원본이고
  백엔드가 내부 액션 3개를 더해 파생한다. `satisfies`가 드리프트를 막는다.
  `GamePhase`는 남겨뒀다 — 아웃바운드 전용인 데다 Redis 스냅샷에 영속돼 있어
  라이브 마이그레이션이 필요하다.
- **ts-jest는 타입 체크를 하지 않는다.** 테스트가 전부 초록인데 `tsc --noEmit`이
  `TS2741`을 잡았다. 실제 게이트는 타입 체크 쪽이다.
- **contract에 jest 설정을 붙였다.** 루트 `npm run test`가 contract 스펙도 돌린다.

### 작업 중 추가로 나온 것 (PR #5)

- **첫 턴 폴백을 없앴다.** `startPreFlop`은 BB 다음에 액션 가능자가 없으면 SB를
  턴으로 지정했다. 블라인드로 전원이 올인된 경우 액션할 수 없는 사람에게 턴을
  주는 것이었고, 곧 도착하는 타임아웃이 그를 폴드시켰다(N-8이 지적한 두 번째
  위치). 이제 `-1`을 그대로 둔다 — `scheduleTurnTimeout`이 `-1`이면 잡을 걸지
  않으므로 타이머 없이 멈추고, 딜러 강제 진행(T8)이 탈출구가 된다.
- **크래시로 통과하던 테스트가 있었다.** "활성 플레이어가 없으면 시작하지 않는다"가
  수정 전에도 초록이었다 — `players[-1]!`의 TypeError도 `toThrow()`에 잡히기
  때문이다. 프로세스를 죽이는 크래시와 정상 거부가 구분되지 않으므로 메시지를
  대조하도록 조였다.
- **헤즈업 첫 액션이 자연히 맞는다.** 버튼=SB로 배치하면 `bbIdx + 1`부터의 탐색이
  한 바퀴 돌아 버튼에 닿는다. 별도 분기가 필요 없다.

### 작업 중 추가로 나온 것 (PR #7)

- **`SETRANGE`를 쓸 수 없었다.** 좌석 비트맵은 `tournament:{id}:seat` 해시의
  `table:{tableId}` **필드**인데 `SETBIT`/`SETRANGE`는 string 키 명령이고 Redis에
  `HSETRANGE`는 없다. 키를 테이블별로 쪼개면 쓸 수 있지만 좌석 현황 조회의
  `hgetall` 한 번이 여러 번으로 늘어난다 — 평탄화 저장은 의도된 설계다.
  같은 일을 Lua로 했다. 서버에서 원자적으로 실행되니 왕복 RMW가 사라지고
  레이아웃은 그대로다. 티켓의 학습(독립적인 비트 → 락이 아니라 원자 연산)은
  그대로 산다. 이 레이아웃에서의 원자 연산이 Lua일 뿐.
- **T2가 놓친 네 번째 지점을 찾았다.** `payment.service.ts`의 `saveSnapShot`이
  `withTableLock` 없는 유일한 스냅샷 쓰기였다. 좌석 락은 좌석**별**이라 다른
  좌석에 앉는 두 사람을 막지 않는다 — 결제는 됐는데 스냅샷에서 사라지는 유저가

- **유령 착석은 "중간 실패"로 재현되지 않는다.** 원본에서 `saveSnapShot`이

  안 건드린다. 진짜 노출 지점은 **커밋 실패** — 콜백은 전부 돌고 커밋만 실패할

  흉내내는 바람에 수정 전 코드에서도 초록이었고, 그래서 스텁을 고쳤다.
- **비트맵 범위 검사를 넣었다.** 예전엔 배열 인덱스로 써서 `bitmapArray[9] = '1'`이
  길이 10짜리 비트맵을 만들었다. 좌석 현황에 없는 자리가 그려진다.

- **가짜 소켓이 실제 동작을 흉내내게 바꿨다.** `send`가 닫힌 상태에서 던지도록

## T9 — 정산 부기 정리

**범위**: `playsync.service.ts`, `dealer.service.ts`
**프론트 영향**: **있음** — `dbSyncStatus` 표시(플레이어 태블릿 + 딜러 콘솔)와

그리고 **순서가 강제된다** — 재시도(N-6)를 붙이는 순간 비멱등 연산(N-7)이 전부
폭탄이 되므로 멱등이 먼저다.

### 전제: 진실의 원천이 핸드 경계에서 교대한다

핸드가 끝나고 DB 트랜잭션이 성공한 시점에는 **DB가 진실의 원천**이다(Redis의
영속성 옵션보다 우선). 다음 핸드가 시작되는 순간부터는 **Redis 스냅샷이 원천**이다.
`initTable`이 `WAITING`으로 넘기는 지점이 이 교대의 경계다.

현재 코드의 순서는 이미 이 의도대로다.

```ts
const isTxSuccess = await this.playsync.syncTableInventoryToDb(state);
if (!isTxSuccess) throw new Error('DB 동기화 실패');
await new TableEngine(state).initTable();   // 여기서 원천이 Redis로 넘어간다
```

체크포인트 → 검증 → 원천 이양. **거짓말하는 것은 boolean 하나뿐이다.** 그래서
N-6은 정책을 새로 정하는 일이 아니라 이미 있는 정책을 실제로 작동시키는 일이다.

따라서 실패 시 `HAND_END`에 멈추는 것은 버그가 아니라 **올바른 안전 상태**다.
문제는 멈추는 것이 아니라 **거기서 나올 방법이 없다는 것**이다.

**일시적 실패 재시도와 장애 복구는 다르다.** DB 커넥션이 한 번 튀는 것은 서버가
살아 있고 스냅샷도 그대로인 상황이라 재시도로 닫힌다. 서버가 죽었다 사는 경우는
멈춘 시간·레벨 재계산·스냅샷 재구성이 필요한 다른 문제다 → "지금 하지 않는 것" 참조.

**완료 조건**
- [x] 상태 전이가 멱등 — 이중 킥·중복 경로에 카운터가 두 번 줄지 않음
- [x] `syncTableInventoryToDb`가 실제 성공 여부를 반환
- [x] 유한 재시도 + **지수 백오프와 지터**
- [x] 재시도 중임이 **테이블 전원에게** 전파됨
- [x] 재시도가 모두 소진되면 딜러에게 원인이 보이고 재시도 진입점이 있음
- [x] `eliminatePlayer`의 `Promise.all`이 실제로 기다림
- [x] 위 전부 회귀 테스트

### 재시도 설계에서 지켜야 할 것

**1. 테이블 전원에게 알린다.** 딜러만 아는 것으로는 부족하다. 플레이어 태블릿에도
"다음 게임 진행에 잠깐 오류가 있다"가 보여야 한다. 재접속한 단말도 같은 것을
봐야 하므로 별도 이벤트가 아니라 **스냅샷의 필드**로 두는 편이 맞다 — 스냅샷은
Redis에 저장되고 재연결 시 그대로 렌더된다. 기존 브로드캐스트 경로를 그대로 타므로
새 배관도 필요 없다. contract의 아웃바운드 스키마에 필드 추가가 따라온다.

**2. 테이블별 재시도가 동시에 몰리면 안 된다.** DB 장애는 여러 테이블을 한꺼번에
실패시키고, 고정 간격으로 재시도하면 전부 같은 순간에 몰려 이미 힘든 DB를
동기화된 파도로 때린다. 재시도가 장애를 늘리는 전형이다. 지수 백오프에 **지터**를
넣어 테이블끼리 시각을 흩는다.

### 작업 중 추가로 나온 것

**N-6의 원인이 반환문이 아니라 삼항이었다.**

```ts
const success = await this.prisma.$transaction(updates) ? true : false;
```

`$transaction`은 성공하면 **결과 배열**을 돌려준다. 배열은 비어 있어도 truthy라
`success`는 언제나 `true`였다. 실패하면 애초에 throw라 이 줄에 도달하지도 않는다.
fixlist는 "항상 true를 반환한다"는 증상만 적었고 원인은 다른 자리에 있었다.
try/catch로 감싸 실제 성공 여부를 반환하도록 고쳤다.

**재시도를 락 밖에 둘 수 있었던 근거.** `withTableLock`의 TTL은 5000ms인데
백오프를 포함한 재시도는 그보다 길어질 수 있다. 락을 쥔 채 기다리면 TTL이 먼저
만료되고, 그 뒤 해제 시도는 **남의 락을 푸는 것**이 된다. 락 밖으로 뺄 수 있는 것은
`HAND_END`가 정지 상태이기 때문이고, 그게 성립하는 건 T5가 페이즈를 문지기로 만들고
T8이 그 페이즈에서 `act()`를 막았기 때문이다. 둘 중 하나라도 없으면 못 뺀다.

**`currentStack`은 쓰기 전용이다.** 세 곳에서 쓰고 읽는 곳이 없다(T2의
`actionDeadline`과 같은 유형). 즉 체크포인트는 지금 소비자가 없고, 그래서 드리프트가
있어도 아무도 모른다. 체크포인트가 실제로 값을 하기 전에 **복구 경로**가 필요한데,
그건 서버 장애 복구 영역이라 "지금 하지 않는 것"으로 미뤘다.
`buttonUser`가 DB 스키마에 없어 스냅샷 재구성이 애초에 불가능한 것도 같은 맥락.

**contract 아웃바운드 스키마 추가는 없었다.** 위에서 예상했던 것과 달리 테이블
스냅샷은 contract에 스키마가 없고 백엔드 타입(`game-engine/types.ts`)이 그대로
나간다. `dbSyncStatus`는 `TableState`에만 추가했다. 스냅샷 전체를 contract로
올리는 것은 이 티켓 범위가 아니다.

**T7의 `never` 분기가 작동했다.** contract에 `RETRY_CHECKPOINT`를 추가하자
`ws.gateway.ts(291,15)`에서 `TS2322 ... not assignable to type 'never'`가 났고,
케이스를 채울 때까지 컴파일이 막혔다. 게이트웨이에 배선하는 것을 잊을 수 없는 구조.

### 작업 중 추가로 나온 것 (PR #14)

**경로마다 정답이 다르다.** `throw new Error`가 56곳인데 전부 바꾸는 게 아니었다.

| 경로 | 조치 | 이유 |
|---|---|---|
| HTTP | Nest 예외 | 던지는 종류가 그대로 상태코드가 되고, 그게 사용자에게 보이는 안내를 가른다 |
| WS | 메시지를 구분 | `{ event: 'error', data: e.message }`로 나간다. 문자열 자체가 안내다 |
| 부팅 | 평범한 `Error` 유지 | 프로세스를 죽이는 것이 목적이라 상태코드가 없다 |

400과 409를 가른 기준: 요청 형식은 올바른데 **대상의 현재 상태**가 거부 이유면
409다. 프론트가 "지금은 안 된다"로 안내할 수 있어야 한다.

**로그는 삼킨 실패에만 남겼다.** 테스트가 보는 동작은 이미 테스트가 증명하므로
로그를 겹칠 이유가 없다. 엔진의 `console.log`는 삭제 — 프레임워크 무의존이
테스트 속도의 근거라 `Logger`를 넣을 수 없고, 환급은 이미 회귀 테스트가 금액까지
고정하고 있다.

**`saveInitialTableSnapshots`가 `pipeline.exec()`의 결과를 버리고 있었다.**
에러 핸들링이 "선택 사항"이라는 주석과 함께 통째로 주석 처리돼 있었다.
`exec()`은 명령이 실패해도 던지지 않고 `[err, response]` 배열로 돌려주므로,
결과를 안 보면 모든 실패가 성공처럼 보인다 — T9의 `$transaction` 삼항과 같은
유형이다. 그리고 죽은 코드가 아니라 **토너먼트 시작 경로**(`session.service.ts:243`)라,
조용히 넘어가면 그 테이블은 스냅샷 없이 시작하고 딜러는 첫 액션에서 이 PR이
만든 '테이블 상태를 찾을 수 없습니다'를 **이유도 모른 채** 본다.

**좌석 경합 테스트가 바로 초록이었다.** 경합 패자는 이미 좌석 락에서
`ConflictException`을 받고 있었다(T6). 기존 동작을 검사한 것이라 무의미해서,
스텁으로 락을 통과시킨 뒤 DB 백스톱에 실제로 닿는 테스트로 바꿨다.

것인가**의 문제다. 배포 전제(좌석 태블릿, 매장 와이파이, 온라인 확장 없음)를
문서로 확정한 뒤에 결정해야 한다. 전제를 안 적고 고치면 "왜 여기까지만 했나"에
답할 수 없고, 반대로 전제만 믿고 안 고치면 그 전제가 코드 어디에도 강제되어 있지
않다는 문제가 남는다.

`docs/threat-model.md`에 적을 것:
- 배포 전제와 신뢰하는 것 / 안 하는 것의 선
- 그 전제가 깨지는 조건 (플레이어 폰이 같은 와이파이, 태블릿 탈옥)
- 전제에 의존하지 않고 서버가 강제하는 항목 목록

## T15 — 사이드팟 증발

**항목**: T13에서 발견 (fixlist에 없음)
**범위**: `table-engine.ts`의 `resolveWinner` · `calculateSidePots`, 딜러 콘솔
**프론트 영향**: 있음 (승자 지명 UX)

**자격자가 아무도 승자로 지명되지 않은 사이드팟은 칩이 증발한다.**

`resolveWinner`는 팟마다 `winnerIds.find(id => pot.relevantPlayerIds.includes(id))`로
승자를 찾고, 못 찾으면 그 팟을 조용히 건너뛴 뒤 `state.pot = 0`으로 지운다.

재현: `short` 100 올인 / `p2` 300 / `p3` 300 → 팟 `[300(셋), 400(p2·p3)]`.
딜러가 `['short']`만 지명하면 **400이 사라진다.** 숏스택이 이겼을 때 1등만 찍고
나머지 둘의 승부를 안 찍는 것은 흔한 조작 실수라 실제 도달 가능한 경로다.

**왜 T9에서 분리했는가**: T9는 "정산 한 번이 두 저장소에 어떻게 안착하는가"이고
이건 엔진 로직과 딜러 지명 UX다. 층이 다르고 프론트가 걸린다.

**왜 그냥 두면 안 되는가**: 승자는 계산되지 않고 딜러가 입력하며, 그 대신 부기는
시스템이 책임진다는 것이 이 프로젝트의 도메인 원칙이다. 카드가 실물이라 증발한
칩을 되돌릴 근거가 테이블 위에 남지 않는다.

**폴백 방향**: 서버가 승부를 추측하면 안 된다. 지명이 비어 있는 팟이 있으면
**거부하고 딜러에게 되묻는 쪽**이 원칙에 맞다. 다만 딜러 콘솔이 "어느 팟이 안
찍혔는지"를 보여줘야 실용적이라 프론트 작업이 따라온다.

**함께 마주칠 지점**: `calculateSidePots`가 `totalContributed > 0`만 보고
`hasFolded`를 안 봐서 폴드한 플레이어가 `relevantPlayerIds`에 남는다. 지금은
딜러가 승자를 지명하므로 무해하지만, 이 목록으로 자격을 판정하는 코드가 생기면
폴드한 사람이 팟을 가져간다.

**주의**: 현재 동작은 T13이 `expect(totalChips(state)).toBe(before - 400)`으로
고정해뒀다. **수정하면 이 테스트가 깨진다 — 깨지는 것이 정상이고 기대값을 갱신한다.**

**완료 조건**
- [x] 지명되지 않은 팟이 있으면 **한 칩도 움직이기 전에** 거부
- [x] 거부 후 딜러가 다시 지명할 수 있음 (페이즈·스냅샷 불변)
- [x] 폴드한 플레이어가 자격자 목록에서 빠짐
- [x] 폴드한 플레이어의 칩은 팟에 남음 (총액 보존)
- [x] 위 전부 회귀 테스트, T13 특성화 테스트 기대값 갱신

### 작업 중 추가로 나온 것 (PR #13)

**"함께 마주칠 지점"이 이미 도달한 지점이었다.** 위에 "이 목록으로 자격을 판정하는
코드가 생기면"이라고 적었는데, `resolveWinner`의
`pot.relevantPlayerIds.includes(id)`가 바로 그 코드다. 폴드한 사람을 딜러가 잘못
찍으면 그대로 지급됐다. 나중의 위험이 아니라 지금의 버그였다.

**증발과 폴드는 한 원인의 두 증상이다.** 사이드팟은 "얼마인가"와 "누가 가져갈 수
있나"를 동시에 정하는데 두 모집단이 다르다 — 금액은 폴드 포함 전원, 자격은 폴드
제외. 배열 하나가 둘을 겸하고 있었다. 그래서 **층을 나누는 기준도 폴드하지 않은
사람의 기여액**이다. 층이란 "여기까지 가져갈 자격이 있다"는 선인데 폴드한 사람에겐
그 선이 없다.

`folded 300 / p2 500 / p3 500`이 이전에는 `900(folded·p2·p3) + 400(p2·p3)`,
이후에는 `1300(p2·p3)`. 총액은 같고 자격만 좁혀진다. 폴드를 참여자에서 통째로
빼면 팟 금액이 줄어 증발이 새로 생기므로, 두 역할을 나눠야 했다.

**자격자가 0인 팟은 만들어질 수 없다.** 처음에 폴백을 설계했다가 철회했다. 층이
생기려면 누가 베팅하고 누가 콜해야 하는데, 콜 없이 폴드로 끝난 베팅은
`refundUncalledBets`가 회수한다. 따라서 한 층의 기여자가 둘 이상이면 그중 최소
한 명은 폴드하지 않은 상태다. **일어날 수 없는 경우에 코드를 붙이면 검증할 수도
없고 나중에 홀덤 규칙으로 오해받는다.**

**마지막 층은 구간으로 자르지 않고 남은 전부를 담는다.** 분기가 아니라 식의
형태라서 `sum(sidePots) === pot`이 구조적으로 성립한다. 방어 코드 없이 증발
경로가 닫힌다.

**`refundUncalledBets`가 검증보다 앞이다.** 환급이 기여액을 낮추고 그 값으로 층이
갈려 자격이 정해진다. 거부 후 재지명해도 두 번 환급되지 않는다 — 환급 후에는
최고액과 차순위가 같아져 `highest > secondHighest`가 거짓이 된다. 회귀 테스트로
고정했다.

**게이트웨이는 손대지 않았다.** 실패를 예외로 올리면 `{ event: 'error' }`로 딜러에게만
가고 브로드캐스트되지 않는다(T7에서 만든 경로). 어느 팟이 비었는지를 메시지에
담는 것으로 끝났다.

**통합 테스트를 수정 뒤에 써서 바로 초록이었다.** `git stash`로 엔진만 되돌려
`Received promise resolved instead of rejected`를 확인하고 복원했다. 엔진이 던지는
것과 그 예외가 **저장 전에** 나가는 것은 다른 문제라 이 테스트가 따로 필요하다.

## T16 — 토너먼트 시작의 조용한 실패

**항목**: T12에서 발견 (fixlist에 없음)
**범위**: `session.service.ts`의 `startSession`
**프론트 영향**: 있음 (시작 실패를 상점 화면이 받아야 한다)

`startSession`이 테이블 초기 상태를 만드는 구간이다.

```ts
let initialState = await this.redis.getSnapShot(t.id);
if (!initialState) return null;          // 스냅샷 없는 테이블은 그냥 빠진다
...
if (validTableStates.length > 0) {       // 전부 빠지면 메타 저장 자체를 건너뛴다
  await this.redis.setTournamentMeta(...);
  await this.redis.saveInitialTableSnapshots(...);
}
```

**스냅샷이 없는 테이블은 조용히 제외되고, 모든 테이블이 그렇더라도
`startSession`은 성공을 반환한다.** 참가자는 앉아 있는데 대회는 시작되지 않은
상태이고, 상점 화면에는 아무 문제도 표시되지 않는다.

그런데 DB 트랜잭션(`startedAt` 기록, 참가자 `PLAYING` 전환)은 **이미 커밋된
뒤다.** 즉 DB상으로는 시작했고 Redis에는 아무것도 없는 상태로 갈린다.

**왜 T12에서 분리했는가**: T12는 에러 타입과 로깅이고 이건 진행 로직이다.
로그를 붙여 "빠졌다"고 남기는 것으로는 안 되고, **어떻게 실패해야 하는가**를
정해야 한다 — 부분 실패를 허용할지, 전부 아니면 아무것도 아닌지, DB 커밋을
Redis 저장 뒤로 미룰지. T9에서 정한 "핸드 경계에서 진실의 원천이 교대한다"와
같은 층의 결정이다.

**함께 볼 것**: `saveInitialTableSnapshots`는 T12에서 실패 시 던지도록 고쳤다.
그 예외를 `startSession`이 지금 아무도 잡지 않으므로, DB는 커밋됐는데 예외가
올라가는 경로가 새로 생겼다. 이 티켓이 그 경계를 함께 정리한다.

## 지금 하지 않는 것

미룬 것이지 못 본 것이 아니다. 순서에 대한 판단은 이렇다.

> 과거 MVP의 목표는 **"돌아가긴 한다"**였다. 지금 단계의 목표는 그것을
> **"완벽하게 돌아간다"**로 바꾸는 것이다. 서버 장애 복구는 그 다음이다.

**로직적 실패의 재시도는 "완벽하게 돌아간다"에 포함된다** — DB 커넥션이 한 번
튀는 것은 서버가 살아 있고 스냅샷도 그대로인 상황이라 T9에서 닫는다. 아래는
서버가 죽었다 사는 경우, 즉 다른 문제다.

### 블라인드 레벨의 정지 시간 미반영

레벨은 `startedAt`과 현재 시각의 차이로 매번 계산된다. **서버가 멈춰 있던 시간을
빼는 기능이 없다.** 장애로 20분 서 있었으면 복구 시 레벨이 그만큼 앞질러 있다.
"실제로 진행된 시간이 얼마인가"를 별도로 들고 있어야 한다.

### 스냅샷 유실 시 테이블 재구성

`TablePlayer`가 `seatPosition`과 `currentStack`을 들고 있어 좌석과 스택은
복구 가능하다. 그러나 **`buttonUser`가 DB 어디에도 없다** — `TableState`에만 있고
Redis 스냅샷에만 산다. 복구하면 버튼 위치를 잃고 다음 핸드 블라인드가 엉뚱한
사람에게 간다. 스키마 변경이 필요하다.

그리고 지금은 `currentStack`을 **읽는 코드가 0이다**(쓰기만 3곳). 체크포인트는
찍히지만 그걸로 되돌리는 경로가 아직 없다.
