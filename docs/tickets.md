# Playsync V2 작업 보드 (tickets)

> 브랜치 하나 = 티켓 하나 = PR 하나.
> 각 티켓은 [fixlist.md](./fixlist.md)의 항목을 참조만 한다. 문제의 상세와 수정
> 코드는 fixlist에 있고, 여기에는 **무엇을 왜 묶었는지와 순서**만 적는다.
>
> fixlist는 심각도 순으로 정렬된 **발견 대장**이고, 이 문서는 **작업 순서**다.
> 둘이 다른 이유는 커플링 때문이다 — 같은 줄을 고치는 항목은 심각도가 달라도
> 한 브랜치에서 함께 처리해야 한다.

## 순서를 정한 기준

1. **커플링 우선** — 같은 함수/같은 줄을 고치는 항목은 반드시 한 티켓.
2. **의존성** — 다른 티켓의 산출물을 쓰는 티켓은 뒤로.
3. **그다음 심각도** — 위 둘이 같으면 P0 → P1 → P2 → P3.
4. **첫 티켓은 인프라 없이 도는 것으로** — 브랜치/테스트/PR/CI 루프를 처음
   도는 데 Redis·DB 셋업이 끼면 루프가 아니라 환경 씨름이 된다.

## 테스트 방침

fixlist의 [P3-4]는 티켓이 아니라 **다른 티켓들에 녹는다**. 거기 나열된 케이스 9개 중
6개가 T1/T4의 회귀 테스트이므로, 각 티켓에서 실패하는 테스트를 먼저 쓰고 고치면
자연히 생긴다. T13에는 어디에도 안 붙는 케이스(사이드팟, 환급)만 남는다.

## T2 — 게임 상태 동시성

**항목**: P0-4, P1-3
**범위**: `redis.service.ts`(락 헬퍼 추가), `playsync.service.ts`, `dealer.service.ts`
**프론트 영향**: 없음

`getSnapShot → 수정 → saveSnapShot`이 락 없이 4곳에서 동시 실행 가능 (lost update).

**왜 함께 묶는가**: fixlist [P0-4] 수정 2번이 "[P1-3] 수정판과 함께 적용"이라고
명시한다. 락으로 감싸는 것과 턴 재검증이 `handleAction` 같은 본문이라 따로 못 고친다.
게다가 [P1-3]에는 **순서 함정**이 있다 — 턴 재검증을 타임아웃 잡 제거보다 **먼저**
해야 한다. 뒤에 두면 다음 플레이어의 타이머를 유실시켜 P1-1 데드락이 다른 경로로
재발한다. fixlist의 원안이 이 결함을 가지고 있었고 2차 검증에서 교체되었다.

**왜 T1 다음인가**: 나머지 티켓이 상태를 만지는 동안 lost update가 살아 있으면
버그의 원인이 내 수정인지 레이스인지 구분이 안 된다. 바닥을 먼저 깐다.

**인지하고 갈 트레이드오프**: `resolveWinners`는 리바인 응답을 최대 15초 기다리는
동안 락을 쥔다. I/O 대기를 락 안에 넣는 것은 원래 안티패턴이다. 지금은 TTL 30초로
넘기고, 리바인 대기를 락 밖으로 빼는 리팩터는 T5 이후 별도로 다룬다.

**완료 조건**
- [x] `withTableLock` 헬퍼 — `SET NX PX` 획득, 토큰 대조 후 Lua로 해제
- [x] 상태를 수정하는 4곳(`handleAction`, `startPreFlop`, `handleDealerAction`, `resolveWinners`) 전부 적용
- [x] 턴 재검증이 타임아웃 잡 제거보다 먼저 실행됨
- [x] KICKED 유저 처리에서 `engine.act` 이중 호출 제거

### 작업 중 추가로 나온 것 (PR #3)

fixlist에 없던 항목들. 락만으로는 안 닫히는 구멍이라 함께 처리했다.

- **`actionDeadline`을 서버가 읽는다.** 백엔드 4곳이 쓰기만 하고 읽는 곳은 프론트
  카운트다운뿐이었다. "30초 지났나"를 판정하는 주체가 BullMQ 잡의 발화 시각밖에
  없었던 셈 — 시간 기반 규칙을 이벤트 도착 순서로 대체한 구조였다. 마감이 지나
  도착한 액션은 도착 순서와 무관하게 `TIME_OUT`으로 처리한다.
- **타이머 세대(`timerEpoch`).** `jobId`가 `tableId` 고정이라, 제거에 실패한 상태에서
  `add`하면 BullMQ가 조용히 무시하고 그 잡이 `removeOnComplete`로 사라져 아무도
  타이머가 없는 테이블이 남았다. 진행 중인 잡은 지울 수 없고 BullMQ는 at-least-once라
  제거 성공에 의존하는 구조 자체가 틀렸다. `jobId`에 세대를 붙여 충돌을 없애고,
  낡은 잡은 세대 불일치로 스스로 폐기되게 했다.
- **`startPreFlop`도 같은 순서 결함이 있었다.** phase를 확인하기 전에 잡부터 지우고
  조기 반환해서, 진행 중인 핸드의 타이머가 통째로 사라졌다.
- **큐 교체를 `scheduleTurnTimeout` 한 곳으로 모았다.** 조기 반환 경로가 큐를 아예
  부를 수 없으므로, 순서가 다시 뒤집히는 것이 주석이 아니라 구조로 막힌다.
- 테이블에 없는 유저(`playerIdx === -1`)를 진입점에서 거부. 잡 delay 31000 /
  deadline 30000 불일치 정리.

**남긴 부채**: `resolveWinners`의 리바인 대기(최대 15초)가 여전히 락 안에 있다.
TTL 30초로 넘겼다. → T5 (PR #6에서 해소, 정산을 3구간으로 분해)

## T3 — WS 인바운드 경계 + contract 도입

**항목**: P0-2, P0-3, P3-2, N-9.3
**범위**: `ws.gateway.ts`, `packages/contract`, `frontend`(액션 전송부)
**프론트 영향**: **있음** — 유일하게 백/프론트 동시 변경

게이트웨이의 인바운드 경계 전체를 한 번에 세운다.

- P0-2: `data.action`을 검증 없이 엔진에 전달. `ActionType`이 숫자 enum이라
  유저가 `TIME_OUT`/`DEALER_KICK`/`DEALER_FOLD`를 직접 보낼 수 있다.
- P0-3: 딜러 JWT의 `tableId`와 접속 쿼리의 `tableId`를 대조하지 않는다.
- N-9.3: 인증된 유저가 아무 `tableId`로나 붙어 상태를 수신한다 (관전).
- P3-2: Origin 미검증. `@WebSocketGateway({ cors })`는 `WsAdapter`에서 무시되므로
  핸드셰이크의 `Origin` 헤더를 직접 봐야 한다.

**왜 함께 묶는가**: 넷 다 `handleConnection` / `handlePlayerAction` 두 함수에 들어간다.
따로 하면 같은 함수를 네 번 고치고 그때마다 프론트 회귀 확인을 반복한다.

**왜 contract가 여기서 도입되는가**: P0-2를 화이트리스트 배열로 막으면 액션이
추가될 때 배열 갱신을 잊는 순간 다시 뚫린다. contract에 클라가 보낼 수 있는 액션만
정의하면 **타입이 곧 화이트리스트**가 되어 그 실수가 불가능해진다. 즉 이 티켓에서
`packages/contract`의 첫 실제 내용이 채워진다.

**숫자 enum → 문자열**: 같이 한다. 와이어에 `4` 대신 `"CALL"`이 흐르면 로그가 읽히고
오프바이원 한 칸으로 다른 액션이 되는 버그 클래스가 사라진다. 프론트 액션 전송부도
같이 바뀐다 — UI/UX는 나중에 갈아엎지만 **이 변경은 로직이라 그대로 남는다.**

**완료 조건**
- [x] `PlayerActionSchema` (`.strict()`) — CHECK/CALL/FOLD/RAISE만 존재
- [x] 딜러 토큰 `tableId` 대조
- [x] 접속 시 테이블 소속 검증
- [x] Origin 검증, 오해를 부르는 `cors: true` 제거
- [x] 프론트가 contract 스키마로 액션 전송
- [x] `handlePlayerAction`에 role 검사 (딜러 토큰으로 플레이어 액션 차단)

### 작업 중 추가로 나온 것 (PR #4)

- **딜러 경로도 스키마가 필요했다.** `DealerActionSchema`를 함께 정의했다. 모르는
  액션이 오면 switch가 아무 case에도 안 걸려 `updatedState`가 `undefined`인 채로
  테이블 전원에게 브로드캐스트되고 있었다. 스키마 + `if (updatedState)` 가드로 막았다.
- **contract가 즉시 죽은 코드 두 개를 드러냈다.** 프론트가 `token`/`tableId`를
  페이로드에 실어 보내는데 서버는 읽지 않았고(핸드셰이크에서 이미 인증됨),
  ALL-IN TO CALL 버튼이 읽히지 않는 `amount`를 보내고 있었다. `.strict()`가
  이것들을 에러로 만들어서 지웠다 — 리팩터가 아니라 검증이 찾아낸 것.
- **`ActionType` 숫자 enum → 문자열을 엔진까지 관통시켰다.** contract가 원본이고
  백엔드가 내부 액션 3개를 더해 파생한다. `satisfies`가 드리프트를 막는다.
  `GamePhase`는 남겨뒀다 — 아웃바운드 전용인 데다 Redis 스냅샷에 영속돼 있어
  라이브 마이그레이션이 필요하다.
- **ts-jest는 타입 체크를 하지 않는다.** 테스트가 전부 초록인데 `tsc --noEmit`이
  `TS2741`을 잡았다. 실제 게이트는 타입 체크 쪽이다.
- **contract에 jest 설정을 붙였다.** 루트 `npm run test`가 contract 스펙도 돌린다.

### 작업 중 추가로 나온 것 (PR #5)

- **첫 턴 폴백을 없앴다.** `startPreFlop`은 BB 다음에 액션 가능자가 없으면 SB를
  턴으로 지정했다. 블라인드로 전원이 올인된 경우 액션할 수 없는 사람에게 턴을
  주는 것이었고, 곧 도착하는 타임아웃이 그를 폴드시켰다(N-8이 지적한 두 번째
  위치). 이제 `-1`을 그대로 둔다 — `scheduleTurnTimeout`이 `-1`이면 잡을 걸지
  않으므로 타이머 없이 멈추고, 딜러 강제 진행(T8)이 탈출구가 된다.
- **크래시로 통과하던 테스트가 있었다.** "활성 플레이어가 없으면 시작하지 않는다"가
  수정 전에도 초록이었다 — `players[-1]!`의 TypeError도 `toThrow()`에 잡히기
  때문이다. 프로세스를 죽이는 크래시와 정상 거부가 구분되지 않으므로 메시지를
  대조하도록 조였다.
- **헤즈업 첫 액션이 자연히 맞는다.** 버튼=SB로 배치하면 `bbIdx + 1`부터의 탐색이
  한 바퀴 돌아 버튼에 닿는다. 별도 분기가 필요 없다.

## T5 — 리바인 응답 처리

**항목**: P1-2, N-5
**범위**: `playsync.service.ts`의 `processRebuy`, `table-engine.ts`의 `handleHandEnd`
**프론트 영향**: 없음

- P1-2: 15초 타임아웃으로 resolve된 경우 `once` 리스너를 제거하지 않아 stale
  리스너가 무한 누적된다. `new Promise(async ...)` 안티패턴도 함께.
- N-5: 리바인 성공 브로드캐스트가 엔진의 스택 반영보다 **먼저** 나가서, 전파된
  상태에는 리바인 스택이 아직 0이다.

**왜 함께 묶는가**: 둘 다 `processRebuy`의 같은 Promise 블록이다. N-5를 제대로
고치려면 엔진 콜백 시그니처를 건드려야 하는데, P1-2가 그 블록을 재작성하는
김에 같이 하는 게 맞다.

**주의**: P1-2는 1차 리뷰에서 **오진**이었던 항목이다. "응답이 씹힌다"고 진단했으나
`emit`은 리스너를 전부 호출하므로 재현되지 않았고, 실제 문제는 누수였다. 수정
코드는 유효하지만 **왜 원래 진단이 틀렸는지**가 이 티켓의 학습 포인트다.

**완료 조건**
- [x] 타임아웃 경로에서 `removeListener` 호출
- [x] executor에서 `async` 제거
- [x] 브로드캐스트가 스택 반영 이후에 나감

### 작업 중 추가로 나온 것 (PR #6)

- **T2가 넘긴 락 부채를 여기서 갚았다.** `resolveWinners`가 리바인 응답을 최대
  15초 기다리는 동안 테이블 락을 쥐고 있었다(그래서 TTL을 30초로 늘려야 했다).
  세 구간으로 쪼갰다 — 팟 분배(락) → 리바인 대기(**락 밖**) → 탈락 확정·초기화(락).

  구조적으로 사라진다 — 반영과 전파가 같은 락 안에 있다.
- **3단계는 스냅샷을 다시 읽는다.** 2단계 동안 각 리바인이 자기 락으로 스택을

- **엔진에서 `rebuyCallback`을 걷어냈다.** 엔진이 콜백으로 서비스의 15초 I/O를
  자기 안으로 끌어들이던 구조였다. 지금은 `resolveWinner`(정산) / `applyRebuy`
  (반영) / `initTable`(WAITING 복귀)로 나뉘고, 대기는 호출자 몫이다.
- **`setTournamentDashboard`가 아무도 못 읽는 형태로 쓰고 있었다.** 대시보드는

## T6 — 좌석 정합성

**항목**: P2-1, N-3

**연결되는 학습**: N-3은 `SETBIT`/`SETRANGE`로 read-modify-write 자체를 없앤다.
T2의 락과 대비된다 — 비트처럼 독립적인 것은 원자 연산이 맞고, 게임 상태처럼
필드 간 일관성이 필요한 것은 락이 맞다. 이 경계가 이 프로젝트의 핵심 학습이다.

**완료 조건**
- [x] Redis 쓰기가 `$transaction` 밖(커밋 후)으로 이동
- [x] 착석 스냅샷 수정이 `withTableLock` 안에서 일어남
- [x] 좌석 비트맵 수정이 원자적
- [x] 위 전부 회귀 테스트

### 작업 중 추가로 나온 것 (PR #7)

- **`SETRANGE`를 쓸 수 없었다.** 좌석 비트맵은 `tournament:{id}:seat` 해시의
  `table:{tableId}` **필드**인데 `SETBIT`/`SETRANGE`는 string 키 명령이고 Redis에
  `HSETRANGE`는 없다. 키를 테이블별로 쪼개면 쓸 수 있지만 좌석 현황 조회의
  `hgetall` 한 번이 여러 번으로 늘어난다 — 평탄화 저장은 의도된 설계다.
  같은 일을 Lua로 했다. 서버에서 원자적으로 실행되니 왕복 RMW가 사라지고
  레이아웃은 그대로다. 티켓의 학습(독립적인 비트 → 락이 아니라 원자 연산)은
  그대로 산다. 이 레이아웃에서의 원자 연산이 Lua일 뿐.
- **T2가 놓친 네 번째 지점을 찾았다.** `payment.service.ts`의 `saveSnapShot`이
  `withTableLock` 없는 유일한 스냅샷 쓰기였다. 좌석 락은 좌석**별**이라 다른
  좌석에 앉는 두 사람을 막지 않는다 — 결제는 됐는데 스냅샷에서 사라지는 유저가
  생긴다. T2 때 dealer/playsync만 훑고 payment을 빠뜨렸다.
- **유령 착석은 "중간 실패"로 재현되지 않는다.** 원본에서 `saveSnapShot`이
  트랜잭션 콜백의 **마지막** 문장이라, 그 앞 DB 쓰기가 실패하면 Redis는 아예
  안 건드린다. 진짜 노출 지점은 **커밋 실패** — 콜백은 전부 돌고 커밋만 실패할
  때, 그때 쓴 스냅샷만 롤백되지 않고 살아남는다. 첫 테스트가 중간 실패를
  흉내내는 바람에 수정 전 코드에서도 초록이었고, 그래서 스텁을 고쳤다.
- **비트맵 범위 검사를 넣었다.** 예전엔 배열 인덱스로 써서 `bitmapArray[9] = '1'`이
  길이 10짜리 비트맵을 만들었다. 좌석 현황에 없는 자리가 그려진다.

## T14 — 인증 정책 결정

**항목**: N-9.1, N-9.2
**범위**: `ws.gateway.ts`, `dealer.service.ts`
**프론트 영향**: 있음 (WS 연결 방식이 바뀌면)
**선행 조건**: 정책 결정 필요 — 코드보다 먼저 `docs/threat-model.md`를 써야 한다

- N-9.1: WS 토큰이 URL 쿼리스트링으로 전달되어 액세스 로그에 남는다.
- N-9.2: 딜러 OTP가 4자리인데 시도 횟수 제한이 없다. 1만 회면 뚫린다.

**왜 마지막이고 왜 별개인가**: 이 둘은 "버그"가 아니라 **신뢰 경계를 어디에 그을
것인가**의 문제다. 배포 전제(좌석 태블릿, 매장 와이파이, 온라인 확장 없음)를
문서로 확정한 뒤에 결정해야 한다. 전제를 안 적고 고치면 "왜 여기까지만 했나"에
답할 수 없고, 반대로 전제만 믿고 안 고치면 그 전제가 코드 어디에도 강제되어 있지
않다는 문제가 남는다.

`docs/threat-model.md`에 적을 것:
- 배포 전제와 신뢰하는 것 / 안 하는 것의 선
- 그 전제가 깨지는 조건 (플레이어 폰이 같은 와이파이, 태블릿 탈옥)
- 전제에 의존하지 않고 서버가 강제하는 항목 목록
