# Playsync V2

오프라인 홀덤 토너먼트 운영 시스템.

플레이어는 실제 테이블에 모여 앉고 카드는 사람 딜러가 실물로 딜링한다.
시스템이 맡는 것은 카드를 제외한 나머지 — 칩 스택, 팟, 베팅 순서, 블라인드
레벨, 좌석 배치, 탈락과 리바인, 그리고 상금이다.

> **이 리포지토리의 목적**
> MVP는 "돌아가긴 한다"에서 멈춰 있었다. V2는 그 코드를 옮겨 온 뒤
> 코드 리뷰에서 찾은 문제를 하나씩 고쳐 **"정확하게 돌아간다"**로 만드는 것을
> 목표로 한다. 발견된 문제는 [`fixlist.md`](./fixlist.md), 고친 내역과 판단
> 근거는 [`tickets.md`](./tickets.md)에 있다.

## 1. 도메인 — 카드는 물리, 칩은 디지털

이 한 줄이 코드 곳곳의 설계를 설명한다.

**`TableState`에 덱도, 홀카드도, 커뮤니티 카드도 없다.** 셔플·핸드 랭킹·승자
판정 로직이 없는 것은 누락이 아니다. 그건 테이블 위에서 일어난다.

**승자는 계산되지 않고 입력된다.** 딜러가 실물 카드를 보고 `resolveWinners`에
승자를 넘긴다. 그래서 이 경로에는 "검증할 정답"이 존재하지 않고, 딜러 입력을
신뢰할 수밖에 없다. 대신 **부기는 시스템이 끝까지 책임진다** — 칩 총량, 사이드팟
분배, 상금 지급이 그것이다.

**딜러가 진행의 트리거다.** `startPreFlop`, `handleDealerAction`,
`resolveWinners`가 사람의 클릭에서 출발한다. 즉 딜러 경로와 플레이어 경로가
동시에 같은 상태를 건드린다. 동시성이 예외가 아니라 기본 시나리오인 이유다.

**플레이어 단말은 좌석에 고정된 태블릿이다.** 버튼과 슬라이더만 있다. 다만
그것은 **UI의 제약이지 서버의 제약이 아니다** — 망이 행사장 WiFi라 같은 망의
아무 단말이나 WS 엔드포인트를 직접 열 수 있다. 그래서 권한은 화면이 아니라
게이트웨이에서 본다.

### 되돌릴 수 없는 일은 마지막에

카드가 실물이라 **잘못 나간 지급을 되돌릴 근거가 테이블 위에 남지 않는다.**
그래서 반복해서 같은 형태의 결정을 내렸다.

| 경계 | 먼저 | 나중에 |
|---|---|---|
| 핸드 종료 | Redis 정산 | DB 커밋(체크포인트) |
| 대회 시작 | Redis 준비 | DB `ONGOING` 커밋 |
| 대회 종료 | 상금 정산 확인 | 테이블·Redis 삭제 |
| 팟 분배 | 전체 검증 | 칩 이동 |

되돌릴 수 있는 일을 먼저 하고, 커밋 한 번이 "그 일이 일어난 단일 순간"이 되게
한다. 실패하면 이전 상태로 남아 **다시 누르는 것이 곧 재시도**가 된다.

## 2. 구조

npm workspaces 모노레포.

| 워크스페이스 | 역할 |
|---|---|
| `backend` | NestJS 11. 게임 로직, WebSocket, DB/Redis |
| `frontend` | Next.js **(재구성 예정 — 현재 명세 작성 단계)** |
| `packages/contract` | 백엔드/프론트가 공유하는 zod 스키마. **경계를 넘는 것만** 정의 |

### 백엔드 모듈

| 모듈 | 역할 |
|---|---|
| `game-engine` | 프레임워크 의존성이 없는 순수 상태머신. 베팅 라운드, 사이드팟, 페이즈 전환, 팟 분배 |
| `playsync` | 진행 오케스트레이션. 액션 처리, 타임아웃, 리바인, 탈락, 상금, DB 체크포인트 |
| `dealer` | 딜러 OTP 인증, 딜러 명령(핸드 시작, 폴드/킥, 승자 결정, 체크포인트 재시도) |
| `ws` | WebSocket 게이트웨이. 인바운드 경계와 브로드캐스트 |
| `payment` | 좌석 선점 락, 참가 결제, 좌석 현황 |
| `store` | 상점·대회 관리. 생성, 시작, 종료, 블라인드 구조 |
| `redis` | 상태 스냅샷, 대회 메타, 좌석 비트맵, 유저 컨텍스트, 테이블 락 |
| `auth` / `user` | JWT 인증, 유저·포인트 |
| `scenario` | 여러 이음매를 한 줄로 잇는 시나리오 테스트 (§5) |

### 상태를 어디에 두는가

| | Redis | PostgreSQL |
|---|---|---|
| 성격 | 진행 중인 게임의 **작업 사본** | **진실**. 웹이 읽는 것 |
| 쓰기 빈도 | 액션마다 | 경계 이벤트에만 |
| 담는 것 | 테이블 스냅샷, 대회 메타(전광판), 좌석 비트맵, 락 | 대회·참가·좌석·포인트·거래 내역 |

플레이어 액션 경로에서 DB 쓰기는 없다. 핸드당 수십 번 일어나는 상태 변경은
전부 Redis에서 처리되고, DB에는 핸드 종료·탈락·상금 같은 경계에서만 기록한다.
(딜러 킥은 예외다 — 대회 참가 상태를 바꾸는 일이라 그 자리에서 DB를 만진다.)

### contract 패키지 규칙

- 비밀 값은 **공개형을 contract에 정의**하고 백엔드가 `.extend()`로 내부형을
  만든다. 전체 스키마를 contract에 두고 `.omit()`으로 빼지 않는다 — 프론트가
  import할 수 있게 되는 순간 규칙이 문서로만 남는다.
- 인바운드(클라 → 서버)는 `.strict()`. 모르는 키가 오면 에러.
- 아웃바운드(서버 → 클라)는 zod 기본 스트립. 백엔드에 필드를 추가해도 자동으로
  새지 않는다.
- Prisma 모델과 백엔드 내부 함수 인자는 contract에 넣지 않는다.

## 3. 게임 진행

### 한 핸드

```
WAITING
  ↓ 딜러: START_PRE_FLOP  (이 시점에 블라인드 레벨을 확정한다)
PRE_FLOP → FLOP → TURN → RIVER
  ↓ 액션 가능한 사람이 없으면 지름길
SHOWDOWN                  (차례 없음. 딜러가 승자를 넣는 화면)
  ↓ 딜러: RESOLVE_WINNERS
HAND_END                  (정산·리바인 대기. 다음 핸드의 문지기)
  ↓
WAITING
```

- **블라인드는 핸드 시작 시점에 확정된다.** 진행 중인 핸드의 블라인드가 도중에
  바뀌지 않는다.
- **`SHOWDOWN`에는 차례가 없다**(`currentTurnSeatIndex === -1`). 있으면 딜러가
  승자를 입력하는 동안 남의 태블릿에서 카운트다운이 돈다.
- **`HAND_END`가 문지기다.** 리바인 응답 대기는 최대 15초짜리 사람 입력이라 락
  밖에서 기다린다. 그동안 다음 핸드가 시작되지 않는 것은 `startPreFlop`이
  `WAITING`만 받기 때문이다.

### 블라인드 레벨

저장된 레벨이 진실이 아니라 **`startedAt`과 현재 시각이 진실이다.** 조회할
때마다 다시 계산하고, 바뀌었으면 Redis에 갱신한다.

그래서 서버가 재기동해도 레벨이 되돌아가지 않고, 폴링이 밀리면 한 번에 여러
칸을 건너뛴다. 리바인 마감도 "정확히 그 레벨을 밟는가"가 아니라 "지났는가"로
판정한다 — 건너뛴 대회가 영영 열려 있으면 안 되기 때문이다.

### 사이드팟

층의 경계는 **폴드하지 않은 사람의 기여액**으로만 정해진다. 폴드한 사람은 팟과
핸드를 포기한 것이라 참여자가 아니다. 다만 그가 낸 칩은 테이블에 남아 마지막
층이 흡수한다. 그래서 `sum(sidePots) === pot`이 구조적으로 성립한다.

승자 지명은 **동점 그룹의 배열**로 받는다.

```
[['a','b'], ['c']]   # a·b 공동 1위, c가 3위
```

보드 하이(커뮤니티 카드가 그대로 모두의 최고 핸드)면 살아남은 전원이 한 그룹에
들어간다. 나눠떨어지지 않는 나머지 칩은 **버튼 다음 자리부터 시계 방향으로**
한 칩씩 — 홀덤 표준이고, 버리면 증발이며, 정하지 않으면 딜러마다 달라진다.

지명되지 않은 팟이 하나라도 있으면 **한 칩도 움직이기 전에 거부한다.** 서버가
승부를 추측하면 안 되고, 되묻는 것이 이 도메인에서 유일하게 옳은 처리다.

### 상금

```
프라이즈풀 = totalBuyinAmount (참가비 + 리바인 누적)
분배율     = 대회 생성 시 상점이 입력. 기본값 없음
itmCount   = 분배율 항목 수에서 파생
```

기본 분배율을 두지 않는 이유는 상금 비율이 코드가 정할 수 있는 값이 아니기
때문이다. 대회를 못 만드는 편이 아무도 합의하지 않은 비율로 돈이 나가는 것보다
낫다. `itmCount`를 따로 받지 않는 이유는 "인 더 머니인데 받을 몫이 없는 등수"가
만들어질 수 있어서다.

지급은 `awardPrize` 한 곳에서만 일어난다. 참가자 행·유저 포인트·거래 내역이
한 트랜잭션에서 함께 움직인다. 나머지 원은 1위가 흡수해 `합계 === 풀`을 유지한다.

**대회는 정산이 끝나야 닫힌다.** 닫으면 테이블·딜러 세션·Redis가 지워져 누가 몇
등이었는지 재구성할 수 없다. 게이트는 `걷은 참가비 == 나간 상금`만 보고
**분배율대로인지는 묻지 않는다** — ICM 찹으로 끝나는 대회가 있어 최후 1인이
나오기 전에 관리자가 정산할 수 있어야 하기 때문이다.

> ### ⚠️ 레이크 0% 가정
>
> 현재는 걷은 참가비 **전부**가 상금으로 나간다. 실제 운영에서는 프라이즈풀이
> 참가비의 일부이고 나머지는 상점 몫이어야 한다.
>
> MVP 기준선에서는 그대로 두되, 붙일 때 고칠 자리는 두 군데다.
>
> - `SessionService.completeSession` — `sum(prizeAmount) === totalBuyinAmount`
> - `scenario/full-flow.int-spec.ts` — 참가 전후 포인트 총합 일치
>
> 둘 다 `걷은 것 == 상금 + 상점 몫`으로 바뀐다.

### 탈락과 킥은 다르다

| | 자리 | 칩 | 조작 |
|---|---|---|---|
| **탈락** (스택 0) | 비운다 | 남길 칩이 없다 | — |
| **킥** (딜러가 내보냄) | **남는다** | 앤티·블라인드로 서서히 녹는다 | 무엇을 눌러도 폴드 |

킥된 사람의 칩을 즉시 회수하지 않는 이유는 그 칩이 어디로 갔는지 설명할 수
없기 때문이다. 테이블에 남은 사람들이 가져간다. 조작을 막는 방법이 "연결 차단"이
아니라 "액션을 폴드로 치환"인 이유는 앞서 말한 신뢰 경계 때문이다.

## 4. WebSocket (`/playsync`)

접속은 쿼리스트링의 토큰으로 인증한다.

| 방향 | 이벤트 | 내용 |
|---|---|---|
| → 서버 | `PLAYER_ACTION` | `CHECK` `CALL` `FOLD` `RAISE`(+amount) |
| → 서버 | `DEALER_ACTION` | `START_PRE_FLOP` `RESOLVE_WINNERS` `DEALER_FOLD` `DEALER_KICK` `RETRY_CHECKPOINT` |
| → 서버 | `REBUY_RESPONSE` | 리바인 수락/거절 |
| → 클라 | `renderGame` | 테이블 상태 브로드캐스트 |
| → 클라 | `renderSeatList` | 좌석 현황 (예매 화면) |
| → 클라 | `REBUY_PROMPT` | 리바인 확인 요청 (개별 유저) |

권한은 스키마가 아니라 게이트웨이가 본다.

- **딜러**: 토큰의 `tableId`와 쿼리의 `tableId`를 대조한다. 대조하지 않으면
  A테이블 딜러가 B테이블의 킥·승자 지정 권한을 그대로 얻는다.
- **플레이어**: 서버가 들고 있는 스냅샷에 그 좌석이 있는지 본다. 클라이언트가
  보낸 값을 근거로 삼지 않는다.
- **Origin**: 브라우저 경유 접속만 검사한다(CSWSH). Origin이 없는 접속은
  통과시킨다 — 좌석 태블릿 같은 비브라우저 클라이언트는 이 헤더를 안 보낸다.

`TIME_OUT`처럼 서버 내부에서만 만들어지는 액션은 **스키마에 없어서** 인바운드로
들어올 수 없다. 스키마가 곧 화이트리스트다.

## 5. 테스트

세 계층이고, 각각 다른 종류의 버그를 잡는다.

| | 파일 | 인프라 | 잡는 것 |
|---|---|---|---|
| 단위 | `*.spec.ts` | 없음 | 엔진 같은 순수 로직. 빨라야 TDD 루프가 돈다 |
| 통합 | `*.int-spec.ts` | Redis + PostgreSQL | 락·트랜잭션처럼 **진짜 인프라라야 의미 있는 것** |
| 시나리오 | `scenario/*.int-spec.ts` | 위와 같음 | **이음매**. 부품이 각각 옳은데 조립이 틀린 경우 |

락을 mock으로 테스트하면 검증 대상인 원자성 자체가 사라진다. 그래서 통합
테스트는 `docker-compose.test.yml`이 띄우는 **별도 컨테이너**를 쓴다 — 개발용과
이미지는 같지만 포트(5433/6380)와 저장소가 분리돼 있고 tmpfs라 데이터가 남지
않는다. DB 이름이나 Redis 인덱스로 나누지 않은 이유는, 테스트가 데이터를 지우는
코드라 설정 실수 하나로 개발 DB를 날릴 수 있기 때문이다. **방어 코드보다 구조로
막는다.**

### 시나리오 계층이 왜 따로 필요한가

단위·통합 테스트는 부품을 본다. 그런데 실제로 터진 버그 중에는 **두 함수가 각각
옳은데 순서만 틀린** 것이 있었다(대회 시작이 Redis 준비 전에 DB를 커밋했다).
어느 부품 테스트도 이걸 잡지 못한다.

시나리오는 단계마다 도메인 불변식을 검사한다. 마지막에 한 번만 보면 "어딘가에서
칩이 사라졌다"까지만 알 수 있어서 **틀어진 첫 순간**을 잡도록 만들었다.

```
칩 총량 보존 / 사이드팟 합 == 팟 / 폴드한 사람은 자격 없음
쇼다운에는 차례 없음 / 좌석 비트맵 == 스냅샷
```

현재 시나리오:

| 파일 | 무엇 |
|---|---|
| `full-tournament` | 대회 하나를 끝까지 |
| `dealer-intervention` | 딜러 폴드·킥 |
| `allin-sidepot` | 올인, 사이드팟, 승자 결정 페이즈의 정보 정확도 |
| `elimination-rebuy` | 탈락과 리바인 |
| `timeout` | 제한 시간, 낡은 타이머 |
| `blind-and-prize` | 블라인드 상승 → 등록 마감 → 상금 지급 |
| `board-high` | 동점 분배 |
| `full-flow` | **회원가입부터 대회 마무리까지.** 스텁 없이 WS 경유 |

### 통과한 테스트를 믿지 않는 법

새 테스트가 **처음부터 통과하면 아무것도 증명하지 않는다.** 이 리포에서 실제로
여러 번 겪었다.

- 기존 상수와 같은 값을 입력으로 골라서, 수정 전에도 통과한 테스트
- 이미 다른 계층이 막고 있어서 검증 대상에 닿지도 못한 테스트
- 참가자 테이블만 봐서, 상금이 포인트로 안 나가는데도 초록이던 테스트

그래서 **실패를 먼저 본다.** 버그 수정은 실패하는 테스트로 재현한 뒤 고치고,
사후에 추가한 검사는 제품 코드를 일부러 되돌려 빨간불을 확인한다.

### 베이스라인

```
contract    44
단위       122  (6 suites)
통합       199  (16 suites)
타입 에러     0
```

CI(`.github/workflows/ci.yml`)가 타입 체크·테스트·빌드를 돌린다.

## 6. 실행

### 요구사항

Node.js, Docker.

### 인프라 (PostgreSQL + Redis)

```bash
cd backend && docker-compose up -d
```

### 환경 변수 — `backend/.env`

```env
DATABASE_URL=postgresql://root:<password>@localhost:5432/playsync
DATABASE_PASSWORD=<password>
JWT_SECRET=<secret>
REDIS_HOST=localhost
REDIS_PORT=6379
REDIS_PASSWORD=<password>
```

선택값. **기본값은 전부 운영 기준**이라 설정하지 않아도 안전한 쪽으로 동작한다.

| | 기본 | 용도 |
|---|---|---|
| `MIN_PLAYERS_TO_START` | `6` | 대회 시작 최소 인원 |
| `WS_ALLOWED_ORIGINS` | `http://localhost:3000` | 브라우저 경유 접속 허용 목록 |
| `REBUY_TIMEOUT_MS` | `15000` | 리바인 응답 대기 |

### 명령 (루트에서)

```bash
npm run typecheck      # contract 빌드 후 backend/frontend 타입 체크
npm run build          # contract → backend → frontend
npm run dev:backend    # NestJS watch (http://localhost:3001)
npm run dev:frontend   # Next dev  (http://localhost:3000)
npm run test           # 단위 테스트. 인프라 없이 2초
npm run test:int       # 통합 테스트. 컨테이너 기동부터 자동
```

마이그레이션은 `cd backend && npx prisma migrate dev`.

반복 실행할 때는 `KEEP_TEST_CONTAINERS=1`로 기동을 건너뛸 수 있고
`npm run test:int:down -w backend`로 내린다.

Prisma는 드라이버 어댑터 구성이라 `$disconnect()`가 pg Pool을 닫지 않는다.
테스트에서는 반드시 `closeTestPrisma()`를 쓴다. 아니면 jest가 종료되지 않는다.

`tsc`가 이미 지운 파일의 에러를 계속 보고하면 `.tsbuildinfo`가 낡은 것이다.
`incremental: true`라서 생기는 일이니 `backend/dist`를 지우고 다시 돌린다.

## 7. 지금 없는 것

없다는 사실 자체가 정보다. "곧 붙일 것"과 "일부러 안 한 것"을 구분해 둔다.

### 다음 단계

| | 왜 지금이 아닌가 |
|---|---|
| **서버 장애 복구** | Redis AOF는 켜져 있지만 **되살리는 로직이 없다.** 스냅샷 재구성은 `buttonUser`가 스키마에 없어 막혀 있다 |
| **인증 정책 정리** | WS 토큰이 쿼리스트링에 있고 딜러 OTP가 4자리에 시도 제한이 없다. 위협 모델 문서가 먼저다 |
| **프론트 재구성** | 게임 화면·상점 관리·통합 어드민 전부. 기능 명세 → API 재구성 → 화면 순서 |
| **레이크** | 위 §3 참고 |

### 범위 밖

- 다중 테이블과 테이블 밸런싱 (`manualMovingPlayer()`는 스텁이다)
- ICM 찹 정산 — 관리자가 금액을 정하는 경로. 지금은 마무리 게이트만 있다
- SIT_AND_GO — 스키마에 enum만 있다
- 실제 PG 결제 — 포인트 차감으로 대체
- 서버 다운 시 블라인드 `startedAt` 보정

### 알려진 한계

- **게이트웨이가 인메모리 Map으로 WS 세션을 들고 있다.** 서버를 늘리면
  브로드캐스트가 갈라진다. 단일 프로세스 전제다.
- 사이드팟 목록은 페이즈 전환 때 다시 계산된다. 라운드 중간의 폴드는 다음
  전환까지 목록에 반영되지 않는다. **지급에는 영향이 없고**(정산이 다시 계산한다)
  딜러가 승자를 지명하는 쇼다운 시점에는 정확하다.

## 8. 작업 규칙

- **`main`에서 직접 작업하지 않는다.** 티켓 단위로 브랜치를 딴다.
- PR 제목과 본문은 한국어. 커밋 메시지·주석·문서는 기존 파일의 언어를 따른다.
- 완료를 주장하기 전에 **실제로 명령을 실행하고 출력을 확인한다.**
- 버그 수정은 **실패하는 테스트로 문제를 재현한 뒤** 고친다.
- 머지 후 `git fetch --prune`과 로컬 브랜치 삭제까지 해서 정리한다.
