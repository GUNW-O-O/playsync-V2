## 우선순위

순서를 정한 기준은 1단계와 같다 — **커플링 먼저, 그다음 의존성, 그다음 심각도.**
여기에 하나가 더 붙는다: **프론트를 막고 있는 것을 먼저.** 프론트 재구성이
백엔드 API에 의존하므로, API 형태를 바꿀 작업은 프론트보다 앞서야 한다.

| # | 항목 | 왜 이 순서인가 |
|---|---|---|
| B1 | 인증 정책 결정 | API 형태를 바꾼다. 프론트보다 먼저여야 재작업이 없다 |
| B2 | 서버 장애 복구 | 스키마 변경이 필요해 일찍 해야 마이그레이션이 겹치지 않는다 |
| B3 | 레이크와 상점 정산 | 돈의 규칙. 지금 두 곳에 0% 가정이 박혀 있다 |
| B4 | ICM 찹 정산 | B3의 정산 구조 위에 얹힌다 |
| B5 | 프론트 기능 명세 | B1~B4가 API를 확정한 뒤 |
| B6 | 백엔드 API 재구성 | 명세가 요구하는 형태로 |
| B7 | 프론트 재구성 | 게임 화면, 상점 콘솔, 통합 어드민 |
| B8 | 다중 테이블·밸런싱 | 단일 테이블 전제를 깬다. MVP 밖이라 뒤 |
| B9 | WS 수평 확장 | 트래픽이 실제로 문제가 될 때 |

## B1 — 인증 정책 결정

**전신**: 1단계의 T14 (보류)
**선행**: `docs/threat-model.md` 작성

두 가지가 걸려 있다.

- **WS 토큰이 쿼리스트링에 있다.** 서버 로그·프록시 로그·브라우저 히스토리에
  남는다. 헤더나 첫 메시지로 옮길지, 단명 티켓으로 교환할지 결정이 필요하다.
- **딜러 OTP가 4자리이고 시도 제한이 없다.** 1만 가지를 그냥 다 넣어볼 수 있다.

**먼저 위협 모델을 써야 하는 이유**: 이 시스템의 신뢰 경계가 특이하다. 망이
행사장 WiFi고 태블릿이 좌석에 고정돼 있다. "같은 망의 단말은 이미 반쯤 신뢰된
존재인가, 아닌가"에 따라 두 문제의 답이 달라진다. 정책 없이 코드를 고치면
근거 없는 강화가 된다.

## B2 — 서버 장애 복구

**전신**: `tickets.md`의 "지금 하지 않는 것"

Redis AOF는 켜져 있지만 **되살리는 로직이 없다.** 세 갈래다.

**스냅샷 유실 시 재구성** — `TablePlayer`가 `seatPosition`과 `currentStack`을
들고 있어 좌석과 스택은 복구 가능하다. 그러나 **`buttonUser`가 DB 어디에도
없다.** 복구하면 버튼 위치를 잃고 다음 핸드 블라인드가 엉뚱한 사람에게 간다.
**스키마 변경이 필요하다.**

그리고 지금 `currentStack`은 **읽는 코드가 0이다**(쓰기만 3곳). 체크포인트는
찍히는데 그걸로 되돌리는 경로가 없다.

**블라인드 정지 시간 보정** — 레벨은 `startedAt`과 현재 시각의 차이로 계산된다.
장애로 20분 서 있었으면 복구 시 레벨이 그만큼 앞질러 있다. "실제로 진행된
시간"을 따로 들고 있어야 한다.

**운영자 조작 화면** — 위 둘을 사람이 판단해 조정한다. 정지 시간 보정, 레벨
수동 지정.

## B3 — 레이크와 상점 정산

지금은 **걷은 참가비 전부가 상금으로 나간다.** 실제 운영에서는 프라이즈풀이
참가비의 일부이고 나머지는 상점 몫이다.

고칠 자리가 정확히 두 곳이다. 둘 다 `걷은 것 == 상금 + 상점 몫`으로 바뀐다.

```
SessionService.completeSession        sum(prizeAmount) === totalBuyinAmount
scenario/full-flow.int-spec.ts        참가 전후 포인트 총합 일치
```

**정해야 할 것**: 레이크를 대회 생성 시 비율로 받을지 상점 단위 설정으로 둘지,
상점 몫을 어디에 기록할지(상점 잔고 모델이 없다), 참가비에서 먼저 떼고 남은
것을 프라이즈풀로 볼지.

## B4 — ICM 찹 정산

관리자가 마무리를 수동으로 하는 것은 **설계다.** 최후 1인이 나오기 전에 남은
사람들이 딜로 끝낼 수 있고, 그때 관리자가 금액을 정한다.

지금은 마무리 **게이트만** 있다 — 합이 맞으면 닫힌다. 관리자가 금액을 지정해
지급하는 경로가 없다. `awardPrize`를 그대로 재사용할 수 있는 모양으로 만들어
뒀으므로, 필요한 것은 그것을 부르는 관리자 엔드포인트와 화면이다.

**돈이 나가는 지점은 계속 하나여야 한다.** 별도 지급 경로를 만들면 멱등성과
거래 내역을 양쪽에 구현해야 하고, 한쪽만 고치면 어긋난다.

## B5 — 프론트 기능 명세

프론트는 전면 재구성이다. 화면이 셋이다.

| 화면 | 쓰는 사람 |
|---|---|
| 게임 화면 | 좌석 태블릿. 버튼과 슬라이더뿐 |
| 딜러 콘솔 | 딜러. 핸드 시작, 폴드/킥, 승자 결정 |
| 상점 콘솔 | 상점 운영자. 대회 생성, OTP 확인, 시작/마무리, 전광판 |
| 통합 어드민 | 플랫폼 운영자. 신규 |

명세에 반드시 들어가야 할 것 하나: **승자 결정 화면의 "보드 하이" 버튼.**
누르면 폴드하지 않은 전원을 한 그룹으로 채워 보낸다. 서버에 별도 엔드포인트를
만들지 않는 이유는 돈이 나가는 경로를 하나로 유지하기 위해서다.

## B6 — 백엔드 API 재구성

명세가 나온 뒤. 지금 REST 엔드포인트는 MVP 때 필요한 대로 자란 상태라
정리가 필요하다. contract 패키지의 경계 규칙(인바운드 `.strict()`, 아웃바운드
스트립)을 REST에도 적용할지 여기서 정한다.

## B7 — 프론트 재구성

B5·B6 뒤.

## B8 — 다중 테이블과 밸런싱

`SessionService.manualMovingPlayer()`가 스텁이다(본문이 주석 두 줄).
자리 이동, 테이블 간 인원 균형, 테이블 통합이 여기 들어간다.
**단일 테이블 전제가 깨지므로** 좌석 비트맵과 시나리오 테스트가 함께 바뀐다.

## B9 — WS 수평 확장

게이트웨이가 인메모리 Map으로 세션을 들고 있다.

```ts
private tournamentSessions = new Map<string, Set<WebSocket>>();
private tableSessions = new Map<string, Set<WebSocket>>();
```

서버를 늘리면 브로드캐스트가 갈라진다. Redis pub/sub 등으로 옮기는 작업이지만,
**지금 필요하지 않다** — 한 행사장 한 대회가 단위라 단일 프로세스로 충분하다.
트래픽이 실제로 문제가 될 때 한다.

## 아직 판단하지 않은 것

착수 전에 결정이 필요한 항목들. 코드가 정할 수 없다.

- **대회 취소와 환불.** 지금 `completeSession`은 정산이 끝나야 닫힌다. 그런데
  참가자만 모이고 시작하지 않은 대회를 취소하면 상금이 0이라 게이트에 걸려
  **영영 닫히지 않는다.** 취소는 환불을 동반해야 하고, 환불 경로가 없다
  (`TransactionType.REFUND`는 스키마에만 있다).
- **SIT_AND_GO.** 스키마에 enum만 있다. 인원이 차면 자동 시작하는 모드인데
  규칙이 정의되지 않았다.
- **실제 PG 결제.** 지금은 포인트 차감이 결제를 대신한다.
- **애드온.** 리바인과 다른 규칙인지, 같은 경로로 처리할지.
