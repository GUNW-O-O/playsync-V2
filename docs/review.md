# MVP 완료 시점 코드 리뷰

> 리뷰 범위: `backend/src` 전체 (게임 엔진, 서비스, 게이트웨이, 인프라 계층) + 프로젝트 구조.
> V1 리포지토리가 "구현 완료"로 닫힌 시점의 평가다. **V2는 여기서 출발했다.**
>
> 개별 결함의 문제·위치·수정 방법은 [`fixlist.md`](./fixlist.md)에, 실제로 무엇을
> 어떤 순서로 고쳤는지는 [`tickets.md`](./tickets.md)에 있다. 이 문서는 그 목록을
> 반복하지 않고 **총평과 분포의 원인**만 다룬다.

---

## 1. 총평

도메인(홀덤 토너먼트 운영) 자체의 난이도가 높고, 그 복잡성을 다루기 위한 아키텍처 선택이 목적에 맞게 이루어져 있다. 반면 "정상적인 유저가 정상적인 입력을 보낸다"는 가정(해피패스) 밖의 영역 — 악의적 입력, 동시 요청, 예외 상황 — 은 거의 방어되어 있지 않다.

요약하면 **"돌아가게 만들기"는 달성했고 "깨지지 않게 만들기"가 남은 상태.**

---

## 2. 잘한 점

### 2.1 도메인 난이도와 문제 정의
상태머신 기반 베팅 라운드, 사이드팟, 턴 타이머, 리바인 응답 대기 등 비동기 흐름이 얽힌 시스템. README에 구현 범위와 미구현 범위를 명확히 구분해 둔 점도 좋다.

### 2.2 사이드팟 알고리즘 (`table-engine.ts:108`)
기여도(`totalContributed`) 오름차순 정렬 후 레벨별로 팟을 분리하는 정석 구현. 이 로직을 정확히 짜는 경우가 드문데 올바르게 되어 있다. 콜되지 않은 오버베팅 환급(`refundUncalledBets`)까지 챙겼다.

### 2.3 아키텍처 선택의 적합성
- **Redis** — 수시로 변하는 게임 상태의 스냅샷 저장소. 영속 데이터(PostgreSQL)와 역할 분리가 명확하다.
- **BullMQ 지연 잡 + `jobId`를 테이블 단위로 고정** — 턴이 넘어갈 때 이전 타이머를 덮어쓰기/제거로 관리하는 전략. 도구의 특성을 이해하고 쓴 흔적.
- **EventEmitter `once`** — 리바인처럼 유저 응답을 기다려야 하는 흐름을 이벤트로 풀어 서비스와 게이트웨이의 결합을 낮췄다.

각 도구를 "왜 쓰는지" 설명 가능한 수준으로 선택되어 있다.

### 2.4 동시성 개념의 존재
좌석 선점에 Redis `SET NX` 분산 락, 리바인 결제에 조건부 업데이트(`points: { gte: entryFee }`) + 트랜잭션, 블라인드 레벨을 시각 기반 파생 상태로 설계(`checkAndSyncBlindLevel` — 동시 갱신이 같은 값으로 수렴해 레이스에 강함).

동시성 문제를 인지하고 있다는 증거다. 다만 **적용 범위가 일관되지 않다** — 좌석에는 락을 걸었으면서 정작 핵심인 게임 상태 스냅샷에는 없다.

## 4. 패턴 분석: 왜 이런 분포인가

문제들의 공통점은 전부 **"물어보지 않은 시나리오"** 라는 것이다.

- **잘 된 부분** — 사이드팟, BullMQ `jobId` 전략, EventEmitter 흐름, 파생 상태 블라인드 설계. 스스로 문제를 정의하고 질문했을 때 나오는 구조다. LLM 보조 개발에서 **질문의 질이 좋았다**는 증거.
- **깨지는 부분** — 음수 입력, 동시 도달, AFK, 헤즈업 엣지, `undefined` 브로드캐스트. 리뷰어 없이는 질문할 생각 자체를 못 하는 영역이다. 도구가 아니라 경험의 한계.

같은 사람이 짠 코드 안에서 품질이 갈린 경계가 거기다. **모르는 것을 모른다는 사실 자체를 모르는 구간**이 정확히 결함으로 남았다.

### 레이스 컨디션의 위치 법칙

레이스는 "동시 요청이 많은 곳"이 아니라 **"read-modify-write 사이클이 비동기 경계를 걸치는 곳"** 에서 난다.

이 프로젝트가 그 법칙의 교과서적 사례다. 개별 숫자 필드는 전부 원자 연산(`hincrby`, 조건부 update)으로 잘 처리했고, 놓친 것은 **여러 필드가 뭉친 JSON 스냅샷**과 **비트맵 문자열** 둘뿐이다 — 둘 다 정확히 read-modify-write 패턴이다.

바꿔 말하면 이 개발자는 "무엇이 위험한가"를 값의 종류로 판단하고 있었고, **위험한 것은 값이 아니라 접근 패턴**이라는 층위를 아직 안 보고 있었다.

## 5. 이후

이 리뷰에서 나온 27건이 [`fixlist.md`](./fixlist.md)의 발견 대장이 됐고, V2의 1단계
21개 티켓([`tickets.md`](./tickets.md))으로 처리됐다.

리뷰가 제안한 작업 순서(P0 → P1 → P2 → P3)는 **그대로 따르지 않았다.** 같은 함수를
건드리는 항목은 심각도가 달라도 한 브랜치에서 함께 고쳐야 해서 **커플링 → 의존성 →
심각도** 순으로 재배열했다.

그리고 이 리뷰의 제안 자체도 전부 옳지는 않았다 — 적용 전 코드 대조에서 3건이
정정됐다(서술 오류 1, 역효과 버그를 심는 수정안 1, 무효한 수정안 1). 경위는
[`fixlist.md`](./fixlist.md) 상단에 있다.
