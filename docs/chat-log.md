## 1. V2 목표와 진행 전략 확정
- **사용자 요청(요지):** 기존 MVP 레포를 복사해온 상태에서, 에이전트 코딩을 쓰는 첫 백/프론트 프로젝트로 어떤 작업부터 해야 할지 물음.
- **AI 제안:** 목표 3택(A 포트폴리오 강화 / B 실제품 방향 / C 워크플로 학습) 제시. 목표 확정 후 순서 전략 3안 중 "가드레일 우선을 뼈대로, 첫 티켓만 수직 슬라이스로 관통"을 추천.
- **사람 판단:** 목표는 B+C 결합의 "완성도 높은 포트폴리오 프로젝트"로 정의. 기한 없음, 완성도 4축(견고성·운영·도메인 완결·UX) 전부 선택. 전략 추천은 수용하고, 백엔드 고도화하며 API 명세를 쌓고 프론트는 나중에 몰아서 끝내는 순서를 직접 발의.
- **근거:** "이제 MVP 같은 구현능력 자체는 아무 변별력이 없다"는 본인 전제. 학습 흐름이 흐려지지 않도록 컨텍스트 관리 가이드라인이 필요하다고 명시.
- **결과:** 가드레일 우선 + 첫 티켓 관통 방식으로 진행 순서 확정.

## 15. T2 — 타임아웃 동시성 버그 설계 확정
- **사용자 요청(요지):** 락 없이 경쟁시킨 건 의도였는데 순서가 반대로 됐고, 타임아웃이 지나도 늦게 누른 유저 행동이 입력되는 문제를 이해하고 싶다. 타임아웃 프로세서를 이해 없이 썼던 게 컸다고 자인.
- **AI 제안:** 원인은 락 부재가 아니라 `actionDeadline`이 "쓰기 4곳 / 읽기 0곳"인 것 — 시간 규칙을 이벤트 도착 순서로 대체한 게 근본 결함. `turnEpoch` 불필요, `actionDeadline` 강제 하나로 충분. 락+전제 재확인+순서 교정이 T2 범위.
- **추가 질문·문제제기:** "bullMQ 관련 수정이 위주냐", "T2 범위가 어디까지냐"로 이해를 좁힘. "기존 유저 액션은 내 턴 검사를 외부에서 먼저 한다"는 자기 이해 제시 → AI가 코드 추적으로 반박(턴 검사는 엔진 안, 잡 제거보다 뒤)하고 신규 버그(남의 좌석이 현재 턴 유저의 30초를 무한 리셋) 제시 → "생각도 못했다"며 수용.

## 29. T16 — 토너먼트 시작 경계에 체크포인트 원칙 적용

## 32. 시나리오 테스트 계층 도입 — 로드맵 확정
- **사용자 요청(요지):** "장애 없다는 가정하에 유저 로그인부터 착석, 게임 종료, 승자 도출까지 모든 통합 시나리오 테스트가 있어야 하고, 어느 부분에서 터지나를 도출할 수 있어야 한다. 이런 프로젝트에서 실질적으로 테스트를 어떻게 할지 몰라서 수동으로 눌러보는 테스트를 했던 것." 완료 후 순서: 장애 복구 → 프론트 기능명세 → 백엔드 API 재구성 → 프론트. 시나리오에는 게임 종료까지 실제 나오는 상황 전체(딜러 폴드/킥, 올인·사이드팟, 탈락·리바인, 타임아웃)를 포함하고, 딜러 콘솔의 승자 지명 정확도가 최우선임을 재강조.
- **AI 제안:** 크롬 창 6개=WS 클라이언트 6개라 브라우저 불필요. 단계별 `it` 분리 + 매 단계 불변식 검사(칩 총량, 인원, 좌석, 턴 정합)로 "터진 첫 순간"을 잡는 설계. Playwright는 곧 버릴 프론트 대상이라 배제.
- **사람 판단:** 요구 범위·목적·로드맵 순서는 사람이 정함. 구현 설계는 (AI 제안 수용).
- **근거:** 수동 테스트의 시간 비용, 부품별 테스트만으로는 조립 오류를 못 잡는다는 경험.
- **결과:** PR #16 — 시나리오 5종(총 40단계). 첫 실행에서 실버그 발견·수정 — `nextPhase()`로 도달한 쇼다운에 `currentTurnSeatIndex` 잔존. 킥·타임아웃에 대한 AI 가정 오류 3건은 테스트 쪽 수정. `git add -A`가 무관 파일을 쓸어담은 커밋 사고는 AI가 자체 발견·제거 (AI 자체 판단, 사람 반응 없음).

## 36. 정산 완료가 대회 종료의 전제 + 보드하이 칩 재분배 요구
- **사용자 요청(요지):** 상금 정산까지 끝나야 대회가 끝나고 Redis·DB 캐시를 정리할 수 있어야 함. 보드하이 시 딜러가 참여 인원 전체에 알맞은 칩을 재분배하는 로직도 필요. — 두 요구 모두 직접 발의.
- **AI 제안:** T19(단일 지급 지점 `awardPrize` + `sum(prizeAmount) == totalBuyinAmount` 정산 게이트, 정리는 게이트 뒤) / T20(승자 입력 계약을 동점 그룹 배열로, 나머지 칩은 버튼 왼쪽부터) 순서 제안. 현행은 팟마다 한 명이라 보드하이면 먼저 찍힌 사람이 독식하며 칩 총량 불변식으로는 못 잡는다는 진단.
- **사람 판단:** "19,20을 하나 브랜치에서 같이하고 그 이후 진짜 전체 사용플로우를 검증해."
- **근거:** 미정산 상태로 닫으면 안 됨 (요구 자체가 근거).
- **결과:** PR #18 — 상금 실지급(대회를 열 때마다 포인트가 소각되던 버그 수정), 정산 게이트, `winnerGroups` 계약 변경.

## 37. 보드하이 처리 — 무승부 서버 엔드포인트는 만들지 않음
- **추가 질문·문제제기:** 보드하이는 참여자들이 각자 팟에 기여한 만큼 돌려받으면 되는 것 아닌지? 무승부 엔드포인트가 생긴다면? — 사용자 문제제기.
- **AI 제안:** 기여액 환급이면 폴드 칩이 증발하고 부분 동점에서 갈라짐. 돈이 나가는 경로가 둘이 되므로 별도 엔드포인트를 만들지 말고 딜러 콘솔 버튼("보드하이" = 생존 전원 한 그룹)으로 처리 권고. 백엔드 추가 없음.
- **사람 판단:** 명시적 수용 발화 없음 — 직후 머지·다음 티켓으로 진행 (AI 제안, 사람 명시 반응 없음).
- **결과:** 백엔드 무변경, 프론트 기능 명세 항목으로 남김 (명세 작성은 이 세션에 없음).
