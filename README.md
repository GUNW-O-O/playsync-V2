# Playsync V2

오프라인 홀덤 토너먼트 운영 시스템.

플레이어는 실제 테이블에 모여 앉고 카드는 사람 딜러가 실물로 딜링한다.
시스템이 맡는 것은 카드를 제외한 나머지 — 칩 스택, 팟, 베팅 순서, 블라인드
레벨, 좌석 배치, 탈락과 리바인, 그리고 상금이다.

```
TypeScript · NestJS 11 · Next.js · PostgreSQL · Redis · zod contract · npm workspaces
단위 122 · 통합 199 · 시나리오 8 · 타입 에러 0
```

> **이 리포지토리가 하는 일**
> V1 MVP는 "돌아가긴 한다"에서 멈춰 있었다. V2는 그 코드를 옮겨 온 뒤
> **"정확하게 돌아간다"**로 만드는 작업이고, 그 과정 전체를 **에이전트 코딩으로**
> 진행하면서 판단 근거를 문서에 남겼다.
> 코드보다 [`docs/chat-log.md`](./docs/chat-log.md)가 이 프로젝트의 핵심이다.

## 1. 무엇이 달라졌나 — V1 → V2

V1은 혼자 짜고 혼자 눌러보며 완성한 MVP다. V2는 같은 코드를 대상으로
**AI가 리뷰하고, AI가 고치고, 사람이 도메인 판단과 순서를 쥐는** 방식으로 다시
지었다. 작업 환경 자체가 바뀐 것이 결과의 차이를 만들었다.

| | V1 | V2 |
|---|---|---|
| 검증 | 크롬 창 6개 띄워 손으로 눌러보기 | 단위 122 / 통합 199 / 시나리오 8. 실제 Redis·PostgreSQL 컨테이너 |
| 테스트 코드 | `nest g` 스캐폴드 18개. 전부 `toBeDefined`고 경로를 못 잡아 import에서 죽고 있었다 | 버그마다 실패하는 테스트를 먼저 쓴다(TDD) |
| CI | 없음 | 타입 체크 · 테스트 · 빌드 (`.github/workflows/ci.yml`) |
| 코드 리뷰 | 없음 (1인 개발) | `backend/src` 전수 리뷰 → [`review.md`](./docs/review.md) → 27개 발견 |
| 완료 기준 | 해피패스가 돈다 | 악의적 입력·동시 요청·예외 상황에서도 장부가 맞는다 |
| 기록 | 구현 범위 ✅/🚧 목록 | 발견 대장 · 티켓별 판단 근거 · **결정 로그** |

1단계는 **이틀 · PR 20개 · 티켓 21개**로 끝났다.

### 리뷰가 찾은 것

V1 README는 구현 범위를 ✅ 완료 / 🚧 미구현으로 적었다. V2가 한 일은 그 ✅
목록을 실제로 검증한 것이고, 결과는 **"완료"의 정의가 바뀐 것**이다.

| V1이 완료라고 적은 것 | 실제로는 |
|---|---|
| 상태머신 기반 베팅 라운드 | 음수 레이즈로 칩이 생성됐다. 타임아웃된 플레이어를 기다리며 라운드가 안 끝났다 |
| 사이드팟 계산 | 칩이 증발했다 |
| 탈락 처리 및 최후 1인 우승 처리 | 상금이 기록만 되고 실제로 지급되지 않았다 |
| Redis Persistence로 상태 복구 **보장** | 복구가 가능한 구조는 맞지만, 그것을 읽어 되살리는 경로가 없다 |

돌아가는 것은 사실이었다. 다만 **"정상적인 유저가 정상적인 입력을 보낸다"**는
가정 위에서만 그랬다.

### 무엇을 고쳤나

| | 문제 | 고친 것 |
|---|---|---|
| **돈·권한** | 음수 레이즈로 칩이 생성됐다. 클라이언트가 서버 내부용 액션을 보낼 수 있었다. 딜러 토큰의 `tableId`를 대조하지 않아 A테이블 딜러가 B테이블을 조작할 수 있었다 | 입력 검증, contract 스키마를 인바운드 화이트리스트로, 게이트웨이에서 권한 대조 |
| **동시성** | 상태 `읽기 → 수정 → 쓰기`가 락 없이 네 곳에서 돌았다(lost update) | Redis 테이블 락. 딜러와 플레이어가 같은 상태를 동시에 만지는 도메인이라 예외가 아니라 기본 시나리오였다 |
| **게임 멈춤** | 타임아웃된 플레이어를 기다리며 베팅 라운드가 안 끝났다. 전원 올인이면 페이즈를 넘길 주체가 없었다. 헤즈업 블라인드가 잘못 배치됐다 | 라운드 종료 판정 수정, 딜러 개입(폴드·킥) 경로 수정 |
| **장부** | 사이드팟에서 칩이 증발했다. 롤백된 트랜잭션이 좌석만 남겼다. 대회 시작이 Redis 준비 전에 DB를 커밋했다 | 사이드팟 재계산으로 `sum(sidePots) === pot` 보장, 되돌릴 수 없는 일을 마지막으로(§3) |
| **미구현** | 상금 금액이 상수였다. 그리고 더 큰 것 — **`PRIZE` 거래를 쓰는 코드가 0이라 상금이 기록만 되고 실제로 나가지 않았다.** 동점을 표현할 입력 형식도 없었다 | 프라이즈풀에서 금액 파생, 지급을 `awardPrize` 한 곳으로, 정산이 끝나야 닫히는 게이트, 동점 그룹 입력 |

작업 순서는 발견 순서와 다르다. 같은 함수를 건드리는 항목은 심각도가 달라도 한
브랜치에서 함께 고쳐야 해서 **커플링 → 의존성 → 심각도** 순으로 21개 티켓을 짰다.
락을 두 번째로 깐 이유는, 나머지 티켓이 상태를 만지는 동안 lost update가 살아
있으면 버그의 원인이 내 수정인지 레이스인지 구분할 수 없기 때문이다.

발견 대장은 [`docs/fixlist.md`](./docs/fixlist.md), 티켓별 판단 근거는
[`docs/tickets.md`](./docs/tickets.md).

## 2. AI를 어떻게 썼나

코드 대부분을 AI가 썼다. 그 물량을 사람이 줄 단위로 검증하는 것은 불가능하고,
하려고 하지도 않았다. 사람이 쥔 것은 **무엇을 어떤 순서로 진행할지와 도메인
판단** 두 가지다. 나머지 신뢰는 사람 눈이 아니라 구조가 만들게 했다.

판단 과정은 [`docs/chat-log.md`](./docs/chat-log.md)에 38개 항목으로 남아 있다.
각 항목은 **요청 → AI 제안 → 사람 판단 → 근거 → 결과** 형식이고,
**기각되거나 철회된 AI 제안도 그대로 남겼다.** 무엇을 골랐는지보다 무엇을 왜
버렸는지가 같은 자리를 다시 볼 때 쓸모 있기 때문이다.

### 진행을 설계한다

- **가드레일 먼저.** 티켓에 들어가기 전에 CI와 테스트 러너부터 세웠다. 무엇을
  깨뜨려도 잡히는 바닥이 없으면 에이전트의 속도는 부채 생성기가 된다.
- **티켓마다 같은 리듬.** 구현 전에 "이 티켓이 어떤 버그를 고치는가"를 설명받고
  합의한 뒤 착수시켰다. 코드를 읽는 대신 문제를 내 언어로 이해하는 관문이다.
- **병렬은 판단이 필요 없는 것만.** 파일이 겹치지 않는 티켓만 워크트리로 병렬
  위임하고, 설계 결정이 든 티켓은 대화로 진행했다. (chat-log #20)
- **위임에는 제약을 건다.** 기존 동작을 박제하는 특성화 테스트에 걸었던 것은
  **"기대값을 코드에서 역산하지 말고 홀덤 규칙으로 먼저 계산하라"**다. 코드를
  돌려 나온 값을 그대로 적으면 버그까지 정답으로 굳고, 나중에 고치는 사람이
  빨간불을 보고 자기가 틀렸다고 생각한다. 실제로 18건 중 1건이 어긋났고 그게
  사이드팟 칩 증발 버그였다.

### 도메인 판단으로 방향을 잡는다

AI는 코드 전체를 읽지만 **왜 그렇게 짰는지**는 모른다. 버그가 있어도 기존 코드의
의도 자체는 유효한 경우가 많았고, 그때마다 "다른 방향으로 구현하지 말고 내 의도를
정확히 구현하라"로 방향을 잡았다.

- **진실의 원천 — #23.** DB 체크포인트 실패를 AI가 "재시도 큐에 넣고 핸드 진행"
  으로 풀려 했다. 원래 설계(핸드 경계에서 진실의 원천이 DB와 Redis 사이를
  교대한다)를 설명하자 제안이 철회됐고, 기존 코드의 순서가 이미 의도대로였다는
  것이 확인됐다. 고칠 것은 항상 true를 돌려주던 boolean 하나였다.
- **홀덤 규칙 — #27.** AI가 "자격자 없는 사이드팟의 반환 경로"를 만들려 했다.
  홀덤 규칙상 일어날 수 없는 상황이라 기각했다. 불가능한 일에 코드를 붙이면
  검증도 못 하고 나중에 규칙으로 오해받는다.
- **의도의 언어화 자체가 검증이다 — #15.** "턴 검사는 외부에서 하고, 내 턴이면
  잡을 제거한다"는 의도를 설명하자 AI가 코드와 대조해 그 의도가 구현돼 있지
  않다는 것을 찾아냈다. 턴 검사가 엔진 안에, 그것도 잡 제거보다 뒤에 있었다.
  남의 좌석이 현재 턴의 타이머를 무한 리셋할 수 있는 신규 버그가 여기서 나왔다.
- **없어도 되는 기능을 만들려 했다 — #21.** 리뷰가 "딜러가 수동으로 페이즈를
  넘기는 버튼(`forceNextPhase`)을 새로 만들라"고 적고 이유를 댔다. 그런 버튼은
  필요하지 않다. 자리를 비운 사람을 처리하는 수단은 이미 `DEALER_FOLD`(턴 스킵)와
  `DEALER_KICK`(영구 이탈)으로 있었고, **AI는 그 두 액션이 무엇을 하는 것인지
  몰랐다.** 기능은 원래 있고 버그만 고치면 된다고 판단하자 티켓이 신규 기능에서
  버그 수정으로 바뀌었고, 프론트 변경도 사라졌다. 실제 버그는 딜러 폴드가 "턴이
  아닌 사람" 분기에만 있던 것이다. 자리를 비운 사람이 정작 자기 차례일 때 아무
  동작도 하지 않았다. 프리플랍 첫 액션자 UTG는 정의상 현재 턴이라, 가장 급한
  경우만 골라서 깨져 있었다.
- **지어낸 서사 자르기 — #21.** AI가 말한 적 없는 화면 동작을 근거로 심각도를
  올렸을 때 근거를 물어 철회시켰다. AI의 설명에는 사실과 그럴듯함이 섞여 온다.
- **설계를 버그로 오인 — #33.** "createSession이 딜러 OTP를 반환하지 않는다"는
  보고는 버그가 아니었다. OTP는 상점 관리 페이지에서 조회하는 것이 정본이다.

### 신뢰는 구조가 만든다

검증도 사람이 아니라 파이프라인이 한다. 대신 그 파이프라인이 거짓 초록을 내지
못하게 하는 규칙을 사람이 정했다.

**적용 전 재검증을 단계로 강제했다.** 전수 리뷰로 18건 대장을 만든 뒤, 적용
전에 전 항목을 코드와 다시 대조하는 2차 검증을 거치게 했다. 15건 중 13건은
정확했고, 오진 3건이 걸러졌으며, 대조 과정에서 신규 9건이 더 나왔다.

| | 무엇이 틀렸나 |
|---|---|
| `P1-2` | 증상 서술이 틀렸다. 재분석 후 심각도를 P1 → P2로 내렸다 |
| `P1-3` | **제안한 수정 자체에 역효과 버그가 있었다.** 그대로 적용하면 타임아웃 타이머를 유실시켜 데드락이 다른 경로로 재발한다. 순서를 재조정한 판으로 교체 |
| `P3-2` | 제안이 무효였다. `WsAdapter`가 `cors` 옵션을 무시한다 |

`P1-3`처럼 그럴듯한 수정이 더 나쁜 버그를 심는 경우가 있어서, 이 단계 없이 리뷰
결과를 적용하는 일 자체가 새로운 위험이다.

**빨간불을 먼저 보게 했다.** 버그 수정은 실패하는 테스트로 문제를 재현한 뒤
고친다. 규칙이 된 건 실제로 세 번 데였기 때문이다. **새로 쓴 테스트가 처음부터
통과한** 경우들:

- 기존 상수와 결과가 같은 입력을 골라서, 수정 전에도 통과했다
- 다른 계층이 이미 막고 있어서 검증 대상에 닿지도 못했다
- 참가자 테이블만 봐서, 상금이 포인트로 안 나가는데도 초록이었다

셋 다 "통과했으니 됐다"로 넘겼으면 그대로 남았을 버그다. 그래서 사후에 추가한
검사는 **제품 코드를 일부러 되돌려 빨간불을 확인**한다. 이틀에 PR 20개가
가능했던 이유는 코드 생성 속도가 아니라 이 분업이다.

## 3. 도메인 — 카드는 물리, 칩은 디지털

이 한 줄이 코드 곳곳의 설계를 설명한다.

**`TableState`에 덱도, 홀카드도, 커뮤니티 카드도 없다.** 셔플·핸드 랭킹·승자
판정 로직이 없는 것은 누락이 아니다. 그건 테이블 위에서 일어난다.

**승자는 계산되지 않고 입력된다.** 딜러가 실물 카드를 보고 `resolveWinners`에
승자를 넘긴다. 그래서 이 경로에는 "검증할 정답"이 존재하지 않고, 딜러 입력을
신뢰할 수밖에 없다. 대신 **장부는 시스템이 끝까지 책임진다** — 칩 총량, 사이드팟
분배, 상금 지급이 그것이다.

**딜러가 진행의 트리거다.** `startPreFlop`, `handleDealerAction`,
`resolveWinners`가 사람의 클릭에서 출발한다. 즉 딜러 경로와 플레이어 경로가
동시에 같은 상태를 건드린다. 동시성이 예외가 아니라 기본 시나리오인 이유다.

**플레이어 단말은 좌석에 고정된 태블릿이다.** 버튼과 슬라이더만 있다. 다만
그것은 **UI의 제약이지 서버의 제약이 아니다.** 망이 행사장 WiFi라 같은 망의
아무 단말이나 WS 엔드포인트를 직접 열 수 있다. 그래서 권한은 화면이 아니라
게이트웨이에서 본다.

### 되돌릴 수 없는 일은 마지막에

카드가 실물이라 **잘못 나간 지급을 되돌릴 근거가 테이블 위에 남지 않는다.**
그래서 반복해서 같은 형태의 결정을 내렸다.

| 경계 | 먼저 | 나중에 |
|---|---|---|
| 핸드 종료 | Redis 정산 | DB 커밋(체크포인트) |
| 대회 시작 | Redis 준비 | DB `ONGOING` 커밋 |
| 대회 종료 | 상금 정산 확인 | 테이블·Redis 삭제 |
| 팟 분배 | 전체 검증 | 칩 이동 |

되돌릴 수 있는 일을 먼저 하고, 커밋 한 번이 "그 일이 일어난 단일 순간"이 되게
한다. 실패하면 이전 상태로 남아 **다시 누르는 것이 곧 재시도**가 된다.

## 4. 온라인 홀덤과 다른 지점

베팅 라운드·사이드팟·블라인드 구조는 홀덤 표준을 따른다. 여기 적는 것은
**사람 딜러와 실물 카드가 섞여서 생긴 것들**뿐이다.

### 사람을 기다리는 상태가 있다

```
WAITING
  ↓ 딜러: START_PRE_FLOP   (여기서 블라인드 레벨이 확정된다)
PRE_FLOP → FLOP → TURN → RIVER
  ↓ 딜러: RESOLVE_WINNERS
HAND_END                   (정산·리바인 대기)
  ↓
WAITING
```

온라인이면 서버가 카드를 돌리고 스스로 다음 페이즈로 넘어간다. 여기서는 **실물
딜링이 끝나야** 다음이다. `WAITING`과 `HAND_END`는 그래서 있는, 시스템이 사람을
기다리는 구간이다.

- **`SHOWDOWN`에는 차례가 없다**(`currentTurnSeatIndex === -1`). 딜러가 실물
  카드를 보고 승자를 입력하는 동안 남의 태블릿에서 카운트다운이 돌면 안 된다.
- **`HAND_END`가 문지기다.** 리바인 응답은 최대 15초짜리 사람 입력이라 락 밖에서
  기다린다. 그동안 다음 핸드가 시작되지 않는 것은 `startPreFlop`이 `WAITING`만
  받기 때문이다.
- **블라인드는 핸드 시작 시점에 확정된다.** 딜링 도중 레벨이 오르면 이미 깔린
  블라인드와 어긋난다.

### 승자는 계산이 아니라 입력이다

딜러가 동점 그룹의 배열로 넣는다. `[['a','b'], ['c']]` = a·b 공동 1위, c가 3위.

검증할 정답이 없으니 입력을 신뢰하는 수밖에 없다. 대신 **장부는 시스템이 지킨다.**

- 지명되지 않은 팟이 하나라도 있으면 **한 칩도 움직이기 전에 거부한다.** 서버가
  승부를 추측하는 것보다 되묻는 것이 옳다.
- 나머지 칩은 버튼 다음 자리부터 시계 방향으로 한 칩씩. 버리면 증발이고, 정하지
  않으면 딜러마다 달라진다.
- 층 경계를 폴드하지 않은 사람의 기여액으로만 잡아 `sum(sidePots) === pot`이
  구조적으로 성립한다.

보드 하이(무승부)에 서버 엔드포인트를 따로 만들지 않은 이유도 같다 — 딜러 콘솔
버튼이 생존 전원을 한 그룹으로 채워 보내면 된다. **돈이 나가는 경로는 하나여야
한다.** (chat-log #37)

### 블라인드 레벨은 시각에서 파생된다

저장된 레벨이 아니라 **`startedAt`과 현재 시각이 진실이다.** 조회할 때마다 다시
계산한다.

행사장에서 몇 시간 도는 대회라 서버 재기동과 폴링 지연이 실제로 일어난다. 파생으로
두면 재기동해도 레벨이 되돌아가지 않고, 밀리면 여러 칸을 한 번에 건너뛴다. 리바인
마감도 "그 레벨을 밟았는가"가 아니라 **"지났는가"**로 판정한다. 건너뛴 대회가
영영 열려 있으면 안 된다.

### 탈락과 킥은 다르다

플레이어가 물리적으로 그 자리에 앉아 있기 때문에 생기는 구분이다.

| | 자리 | 칩 | 조작 |
|---|---|---|---|
| **탈락** (스택 0) | 비운다 | 남길 칩이 없다 | — |
| **킥** (딜러가 내보냄) | **남는다** | 앤티·블라인드로 서서히 녹는다 | 무엇을 눌러도 폴드 |

킥된 사람의 칩을 즉시 회수하지 않는 이유는 그 칩이 어디로 갔는지 설명할 수 없기
때문이다. 테이블에 남은 사람들이 가져간다. 조작을 막는 방법이 연결 차단이 아니라
**액션을 폴드로 치환**인 이유는 §3의 신뢰 경계 때문이다. 끊어도 같은 망에서 다시
붙을 수 있다.

### 상금은 상점이 정하고 관리자가 닫는다

프라이즈풀은 `totalBuyinAmount`(참가비 + 리바인 누적)이고, 분배율은 대회 생성 시
상점이 입력한다. **기본값을 두지 않았다.** 상금 비율은 코드가 정할 수 있는 값이
아니라, 대회를 못 만드는 편이 아무도 합의하지 않은 비율로 돈이 나가는 것보다 낫다.

지급은 `awardPrize` 한 곳에서만 일어나고, 참가자 행·포인트·거래 내역이 한
트랜잭션에서 함께 움직인다.

**대회는 정산이 끝나야 닫힌다.** 닫으면 테이블·딜러 세션·Redis가 지워져 누가 몇
등이었는지 재구성할 수 없다. 게이트는 `걷은 참가비 == 나간 상금`만 보고
**분배율대로인지는 묻지 않는다.** 오프라인 대회는 ICM 찹으로 끝나기도 해서,
최후 1인이 나오기 전에 관리자가 금액을 정해 정산할 수 있어야 한다.

> **⚠️ 레이크 0% 가정** — 현재는 걷은 참가비 전부가 상금으로 나간다. 실제
> 운영이면 나머지가 상점 몫이어야 한다. 붙일 자리는 두 곳이고 둘 다
> `걷은 것 == 상금 + 상점 몫`으로 바뀐다 ([`backlog.md`](./docs/backlog.md) B3).

## 5. 구조

npm workspaces 모노레포.

| 워크스페이스 | 역할 |
|---|---|
| `backend` | NestJS 11. 게임 로직, WebSocket, DB/Redis |
| `frontend` | Next.js **(재구성 예정 — 현재 명세 작성 단계)** |
| `packages/contract` | 백엔드/프론트가 공유하는 zod 스키마. **경계를 넘는 것만** 정의 |

### 백엔드 모듈

| 모듈 | 역할 |
|---|---|
| `game-engine` | 프레임워크 의존성이 없는 순수 상태머신. 베팅 라운드, 사이드팟, 페이즈 전환, 팟 분배 |
| `playsync` | 진행 오케스트레이션. 액션 처리, 타임아웃, 리바인, 탈락, 상금, DB 체크포인트 |
| `dealer` | 딜러 OTP 인증, 딜러 명령(핸드 시작, 폴드/킥, 승자 결정, 체크포인트 재시도) |
| `ws` | WebSocket 게이트웨이. 인바운드 경계와 브로드캐스트 |
| `payment` | 좌석 선점 락, 참가 결제, 좌석 현황 |
| `store` | 상점·대회 관리. 생성, 시작, 종료, 블라인드 구조 |
| `redis` | 상태 스냅샷, 대회 메타, 좌석 비트맵, 유저 컨텍스트, 테이블 락 |
| `auth` / `user` | JWT 인증, 회원·포인트 |
| `scenario` | 여러 이음매를 한 줄로 잇는 시나리오 테스트 (§6) |

### 상태를 어디에 두는가

| | Redis | PostgreSQL |
|---|---|---|
| 성격 | 진행 중인 게임의 **작업 사본** | **진실**. 웹이 읽는 것 |
| 쓰기 빈도 | 액션마다 | 경계 이벤트에만 |
| 담는 것 | 테이블 스냅샷, 대회 메타(전광판), 좌석 비트맵, 락 | 대회·참가·좌석·포인트·거래 내역 |

플레이어 액션 경로에서 DB 쓰기는 없다. 핸드당 수십 번 일어나는 상태 변경은
전부 Redis에서 처리되고, DB에는 핸드 종료·탈락·상금 같은 경계에서만 기록한다.
(딜러 킥은 예외다 — 대회 참가 상태를 바꾸는 일이라 그 자리에서 DB를 만진다.)

### WebSocket (`/playsync`)

접속은 쿼리스트링의 토큰으로 인증한다.

| 방향 | 이벤트 | 내용 |
|---|---|---|
| → 서버 | `PLAYER_ACTION` | `CHECK` `CALL` `FOLD` `RAISE`(+amount) |
| → 서버 | `DEALER_ACTION` | `START_PRE_FLOP` `RESOLVE_WINNERS` `DEALER_FOLD` `DEALER_KICK` `RETRY_CHECKPOINT` |
| → 서버 | `REBUY_RESPONSE` | 리바인 수락/거절 |
| → 클라 | `renderGame` | 테이블 상태 브로드캐스트 |
| → 클라 | `renderSeatList` | 좌석 현황 (예매 화면) |
| → 클라 | `REBUY_PROMPT` | 리바인 확인 요청 (개별 플레이어) |

권한은 스키마가 아니라 게이트웨이가 본다.

- **딜러**: 토큰의 `tableId`와 쿼리의 `tableId`를 대조한다. 대조하지 않으면
  A테이블 딜러가 B테이블의 킥·승자 지정 권한을 그대로 얻는다.
- **플레이어**: 서버가 들고 있는 스냅샷에 그 좌석이 있는지 본다. 클라이언트가
  보낸 값을 근거로 삼지 않는다.
- **Origin**: 브라우저 경유 접속만 검사한다(CSWSH). Origin이 없는 접속은
  통과시킨다. 좌석 태블릿 같은 비브라우저 클라이언트는 이 헤더를 안 보낸다.

`TIME_OUT`처럼 서버 내부에서만 만들어지는 액션은 **스키마에 없어서** 인바운드로
들어올 수 없다. 스키마가 곧 화이트리스트다.

## 6. 테스트

세 계층이고, 각각 다른 종류의 버그를 잡는다.

| | 파일 | 인프라 | 잡는 것 |
|---|---|---|---|
| 단위 | `*.spec.ts` | 없음 | 엔진 같은 순수 로직. 빨라야 TDD 루프가 돈다 |
| 통합 | `*.int-spec.ts` | Redis + PostgreSQL | 락·트랜잭션처럼 **진짜 인프라라야 의미 있는 것** |
| 시나리오 | `scenario/*.int-spec.ts` | 위와 같음 | **이음매**. 부품이 각각 옳은데 조립이 틀린 경우 |

락을 mock으로 테스트하면 검증 대상인 원자성 자체가 사라진다. 그래서 통합 테스트는
개발용과 분리된 진짜 Redis·PostgreSQL 컨테이너를 띄운다.

### 시나리오 계층이 왜 따로 필요한가

단위·통합 테스트는 부품을 본다. 그런데 실제로 터진 버그 중에는 **두 함수가 각각
옳은데 순서만 틀린** 것이 있었다(대회 시작이 Redis 준비 전에 DB를 커밋했다).
어느 부품 테스트도 이걸 잡지 못한다.

시나리오는 단계마다 도메인 불변식을 검사한다. 마지막에 한 번만 보면 "어딘가에서
칩이 사라졌다"까지만 알 수 있어서 **틀어진 첫 순간**을 잡도록 만들었다.

```
칩 총량 보존 / 사이드팟 합 == 팟 / 폴드한 사람은 자격 없음
쇼다운에는 차례 없음 / 좌석 비트맵 == 스냅샷
```

| 시나리오 | 무엇 |
|---|---|
| `full-tournament` | 대회 하나를 끝까지 |
| `dealer-intervention` | 딜러 폴드·킥 |
| `allin-sidepot` | 올인, 사이드팟, 승자 결정 페이즈의 정보 정확도 |
| `elimination-rebuy` | 탈락과 리바인 |
| `timeout` | 제한 시간, 낡은 타이머 |
| `blind-and-prize` | 블라인드 상승 → 등록 마감 → 상금 지급 |
| `board-high` | 동점 분배 |
| `full-flow` | **회원가입부터 대회 마무리까지.** 스텁 없이 WS 경유 |

## 7. 실행

Node.js와 Docker가 필요하다.

```bash
cd backend && docker-compose up -d   # PostgreSQL + Redis
npx prisma migrate dev               # 마이그레이션

# 루트에서
npm run dev:backend    # NestJS watch (http://localhost:3001)
npm run dev:frontend   # Next dev     (http://localhost:3000)
npm run test           # 단위 테스트. 인프라 없이 2초
npm run test:int       # 통합 테스트. 컨테이너 기동부터 자동
```

`backend/.env`:

```env
DATABASE_URL=postgresql://root:<password>@localhost:5432/playsync
DATABASE_PASSWORD=<password>
JWT_SECRET=<secret>
REDIS_HOST=localhost
REDIS_PORT=6379
REDIS_PASSWORD=<password>
```

`MIN_PLAYERS_TO_START`(6) · `WS_ALLOWED_ORIGINS`(`http://localhost:3000`) ·
`REBUY_TIMEOUT_MS`(15000)은 선택값이다. **기본값이 전부 운영 기준**이라 설정하지
않아도 안전한 쪽으로 동작한다.

## 8. 지금 없는 것

없다는 사실 자체가 정보다. "곧 붙일 것"과 "일부러 안 한 것"을 구분해 둔다.
전체 목록과 순서는 [`docs/backlog.md`](./docs/backlog.md)(B1~B9)에 있다.

### 다음 단계

| | 왜 지금이 아닌가 |
|---|---|
| **인증 정책 정리** | WS 토큰이 쿼리스트링에 있고 딜러 OTP가 4자리에 시도 제한이 없다. 위협 모델 문서가 먼저다 |
| **서버 장애 복구** | 복구가 가능한 구조는 갖췄다(Redis AOF, DB 체크포인트). **그것을 읽어 되살리는 경로**가 아직 없고, 스냅샷 재구성은 `buttonUser`가 스키마에 없어 막혀 있다 |
| **레이크와 상점 정산** | §4 참고 |
| **프론트 재구성** | 게임 화면·딜러 콘솔·상점 콘솔·통합 어드민 전부. 기능 명세 → API 재구성 → 화면 순서 |

### 범위 밖

- 다중 테이블과 테이블 밸런싱 (`manualMovingPlayer()`는 스텁이다)
- ICM 찹 정산 — 관리자가 금액을 정하는 경로. 지금은 마무리 게이트만 있다
- SIT_AND_GO — 스키마에 enum만 있다
- 실제 PG 결제 — 포인트 차감으로 대체
- 서버 다운 시 블라인드 `startedAt` 보정

### 알려진 한계

- **WS 세션만 게이트웨이의 인메모리 Map에 있다.** 게임 상태는 Redis, 진실은
  DB라 서버 자체는 상태를 들지 않지만, 연결만은 프로세스에 묶인다. 서버를
  늘리면 브로드캐스트가 갈라지므로 현재는 단일 프로세스 전제다.
- 사이드팟 목록은 페이즈 전환 때 다시 계산된다. 라운드 중간의 폴드는 다음
  전환까지 목록에 반영되지 않는다. **지급에는 영향이 없고**(정산이 다시 계산한다)
  딜러가 승자를 지명하는 쇼다운 시점에는 정확하다.
